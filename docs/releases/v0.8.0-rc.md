# Aesynx v0.8.0 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest and GitHub checks
are complete.

## Scope

`v0.8.0` is the first x86_64 exception-handling milestone. It keeps the v0.7
GDT/TSS setup, then installs an Aesynx-owned IDT during early `_start`.

It does not claim full fault decoding, CR2 reporting, interrupt-controller
support, external IRQ delivery, syscall/sysret, userspace privilege
transitions, SMP exception stacks, or recovery from arbitrary faults. Those
belong to later milestones.

## Deliverables

- `aesynx-arch-x86_64` `exceptions` module.
- Private static 256-entry x86_64 IDT.
- IDT entries for breakpoint, page fault, and double fault.
- Double-fault gate configured to use the v0.7 double-fault IST slot.
- Stable Rust assembly stubs for exception entry and `iretq` return from
  breakpoint.
- Breakpoint smoke that returns and lets normal boot continue.
- Opt-in `exception-smoke` kernel feature that deliberately triggers a page
  fault.
- Page-fault handler skeleton that prints a bounded serial diagnostic and halts
  rather than returning to the faulting instruction.
- Early kernel boot phase `exception-setup`.
- Opt-in kernel boot phase `exception-smoke`.
- `[TEST] idt=ok`, `[TEST] exception=ok`, and `[TEST] pagefault=ok` markers.
- v0.8 QEMU image, manifest, and serial-log names.
- QEMU boot, panic-smoke, and exception-smoke gates.

## Security Notes

- IDT setup is isolated to the architecture crate.
- Exception assembly is the only new v0.8 unsafe surface.
- The public exception initialization API is safe; unsafe assembly, static IDT
  mutation, and raw exception-frame reads stay private to
  `aesynx-arch-x86_64::exceptions`.
- Breakpoint handling returns through `iretq`; page-fault and double-fault paths
  halt after bounded diagnostics.
- The page-fault smoke deliberately touches a null pointer only under the
  `exception-smoke` feature.
- QEMU normal, panic-smoke, and exception-smoke paths all verify the IDT marker.
- v0.8 pentest follow-up tightened direct-build path remapping, scheduler
  terminal-state handling, capability grant/derive bounds, BootInfo
  architecture validation, Limine response revision compatibility checks, log
  error naming, and bytecode verifier operand types before tag readiness.
- Limine framebuffer and RSDP pointer-to-address normalization explicitly
  documents intentional provenance loss until framebuffer and ACPI consumers add
  local MMIO/firmware-table unsafe contracts.
- `grant` now strips `GRANT` from transferred capabilities by default, so
  re-delegation requires a future explicit audited API.
- Derived ranged capabilities reject zero-length ranges.
- `Dead` tasks are terminal and reject repeated `Dead -> Dead` transitions.
- The bytecode crate has `VerifiedOffset` and `VerifiedTarget` types that can
  only be created through verifier bounds checks.

## Verification

Required before pentest handoff:

```bash
scripts/checks.sh
scripts/generate-sbom.sh
cargo clippy --target x86_64-unknown-none -p aesynx-kernel --features exception-smoke -- -D warnings
cargo xtask build-kernel
cargo xtask image
cargo xtask qemu
cargo xtask qemu --panic-smoke
cargo xtask qemu --exception-smoke
```

## Known Limitations

- No CR2 fault-address output exists yet.
- No page-fault error-code decoding exists yet.
- No general interrupt-controller setup exists yet.
- No user/kernel privilege transition exists yet.
- No SMP/per-core IDT or TSS exists yet.
- Unhandled vectors are not a complete diagnostic surface yet.

# Aesynx v0.36.0 Release Candidate Notes

`v0.36.0` adds the first core-to-core IPC fabric smoke. The release proves the
shape of pairwise multikernel messaging before AP execution is enabled: core 0
can stamp a Ping for core 1, core 1 can observe it, core 1 can stamp a Pong back
to core 0, sequence numbers advance, and a full SPSC link reports fail-closed
backpressure.

This is still model-backed evidence. Secondary cores do not execute Aesynx Rust
code yet, and the queue indices are not live cross-core atomics. The value of
this release is the fail-closed API shape and QEMU gate that later hardware
delivery must preserve.

## Changes

- Added `aesynx-ipc::fabric` with `PairwiseSpscQueue`, `FabricMessage`, and
  `CorePairPingPong`.
- Pairwise queues record producer and consumer core identity and reject wrong
  producer, wrong consumer, loopback pairs, mismatched message routes,
  zero-capacity queues, empty pops, and full pushes before mutation.
- Ping/Pong messages reuse the existing kernel-stamped `MessageHeader` path so
  source, destination, kind, sequence, and optional reply metadata remain
  kernel-owned rather than caller-forged.
- The fabric smoke records release/acquire publish-observe evidence through the
  existing queue-ordering contract.
- QEMU boot smoke now requires `ipc-pingpong ping_seq=`,
  `backpressure_ok=true`, `release_acquire_ok=true`,
  `pairwise_route_ok=true`, and `[TEST] ipc-pingpong=ok`.
- GitHub Actions now pins `actions/checkout` to the v7 tag object SHA.
- Release artifact names and manifests now identify the candidate as
  `v0.36.0`.

## Security Notes

- Backpressure is fail-closed: a full pairwise queue returns
  `FabricError::Backpressure` instead of overwriting an unread message.
- Route checks compare the stamped header source and destination against the
  pairwise link before enqueue.
- Sequence allocation uses checked arithmetic and returns
  `FabricError::SequenceOverflow` on wrap.
- Debug-sensitive payload handling continues to use the existing redacted
  `MessagePayload` and `MessageHeader` behavior.
- The new fabric model is explicitly non-`Sync` by construction and remains
  an `&mut self` model until AP execution and hardware-backed atomic queues are
  implemented.

## Non-Goals

- No secondary core executes Aesynx code.
- No APIC IPI delivery path.
- No live shared-memory queue between hardware cores.
- No cross-core capability transfer yet.
- No global revocation protocol over IPC yet.

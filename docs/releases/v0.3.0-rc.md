## Scope

`v0.3.0` is the QEMU image-skeleton milestone. It creates a generated raw image
and verifies that QEMU can boot far enough to receive deterministic serial
output from a temporary Aesynx stage-0 probe.

It does not claim a Rust kernel entry point, Limine/UEFI handoff, BootInfo
normalization, or first kernel serial output. Those start in `v0.4.0` and later
milestones.

## Deliverables

- `cargo xtask image`
- `cargo xtask qemu`
- Generated raw image under `build/qemu/`
- Generated image manifest under `build/qemu/`
- QEMU serial marker validation


## Pentest Follow-Up Hardening

The v0.3 implementation also tightens security contracts found during the
temporary pentest handoff:

- `KernelImageInfo` fields are private and debug output is redacted to avoid
  KASLR-sensitive address disclosure.
- `IpiVector` rejects CPU-reserved exception vectors and the APIC-sensitive
  `0xff` vector.
- Task identity, owner core, priority, and budget are private and read through
  accessors.
- Device identity, bus, owner core, and state are private and read through
  accessors.
- Task telemetry counters are append-only and exposed through snapshots.
- Core telemetry snapshots are explicitly advisory per-counter samples rather
  than coherent multi-counter transactions.
- The stage-0 boot sector assembler rejects oversized payloads, out-of-range
  branches, and out-of-bounds patches instead of silently truncating or
  panicking.

## Known Limitations

- The image boots a temporary stage-0 probe, not the Rust kernel.
- No `_start` entry exists yet.
- No real bootloader protocol handoff exists yet.
- No BootInfo parsing exists yet.
- No QEMU serial-expect test harness exists beyond the `xtask qemu` smoke.

# Aesynx v0.28.0 Release Candidate Notes

Status: implementation candidate; pending pentest and GitHub checks before tag.

## Summary

`v0.28.0` adds the first local cooperative executor on top of the v0.27 task
model. The executor owns a fixed-capacity local run queue, a fixed-capacity
timer wait queue, and at most one current task. It can dispatch the next
runnable task, yield the current task back to the run queue, move the current
task into timer sleep, and wake one timer waiter back to runnable state.

The kernel boot smoke now verifies an interleaved A, B, A, B task sequence with
one sleep/wake cycle before reporting `[TEST] cooperative-sched=ok`.

## Added

- Added `aesynx-sched::executor` as a separate no_std module so executor logic
  does not grow inside `queue.rs` or `task.rs`.
- Added `LocalExecutor` with deterministic local round-robin dispatch.
- Added executor operations for spawn, dispatch, yield, timer sleep, and timer
  wake.
- Added `ExecutorStatus` and `ExecutorError` so callers can inspect aggregate
  executor state without exposing queue internals or raw task storage.
- Added a kernel cooperative-scheduler smoke that runs two task objects through
  an interleaved sequence, sleeps task B on the timer wait queue, wakes it, and
  verifies final queue state.
- Added QEMU marker coverage for `cooperative-sched task_a=` and
  `[TEST] cooperative-sched=ok`.

## Security And Correctness

- The executor preserves linear task ownership across success and error paths:
  failed yield, failed sleep, failed wake reinsertion, and failed dispatch
  restoration either restore the current task or report an explicit restore
  failure.
- Nested dispatch is rejected before mutating executor state.
- Timer sleep preflights duplicate and full wait-queue conditions before moving
  the current task into `WaitingOnTimer`.
- Timer wake preflights duplicate and full run-queue conditions before removing
  a task from the timer wait queue.
- Executor counters use checked arithmetic and fail closed on overflow.
- The executor is still built on non-`Sync` queue types; shared static or SMP
  use still requires a future IRQ-safe/per-core lock wrapper.
- Debug output remains aggregate-only through `ExecutorStatus`; task IDs are
  visible through explicit status fields for smoke verification, not through raw
  queue storage.
- Pentest follow-up made kernel heap free-list offset decoding release-checked
  with an explicit `CorruptFreeList` error instead of relying on a debug-only
  assertion and release overflow checks as the final backstop.
- Pentest follow-up made `CorruptFreeList` observable through aggregate heap
  telemetry and the QEMU smoke line as `corrupt_free_list_detected=false` during
  normal boot, preserving visibility after the fail-soft error handling change.
- Pentest follow-up recorded that slab double-free membership checks and
  free-page/run searches still use bounded linear scans under the IRQ-masked
  heap lock; O(1) membership/free-page tracking remains required before the heap
  grows materially or scheduler latency depends on tight interrupt windows.
- Pentest follow-up recorded that service-queue release/acquire evidence is
  metadata for the current `&mut self` queue model. A future shared-memory or
  cross-core ring must prove the release/acquire pair on actual atomic
  head/tail or slot-validity updates.
- Pentest follow-up recorded that capabilities contain no secret bytes today,
  but any future secret-bearing capability variant must scrub backing slot
  storage on revoke/delete rather than relying on `Option::None` assignment.

## Known Limitations

- This is cooperative model execution, not architectural context switching.
- There is no saved register frame, task kernel stack switching, preemption,
  timeslice accounting, syscall ABI, userspace task object, or SMP scheduler
  yet.
- Sleep/wake is model-level timer waiting only. It is not wired to production
  timer interrupts or real task blocking.
- Queue membership checks are still linear scans. Large or syscall-hot queues
  need indexed membership tracking before this becomes a live fast path.
- Queue mutation is not wrapped in a live IRQ/preemption guard yet. The queue
  types remain non-`Sync` until a lock wrapper exists, and real executor use
  must protect local queue mutation from interrupt re-entry.

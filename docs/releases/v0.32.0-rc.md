# Aesynx v0.32.0 Release Candidate Notes

`v0.32.0` prepares Aesynx for future AI-assisted policy without loading,
executing, or trusting an AI model.

## What Changed

- Split `aesynx-ai-policy` into focused no_std modules for confidence, policy
  decisions, model manifests, and scheduler features.
- Added schema-v1 `ModelObjectManifest` with model ID, kind, policy domain,
  input/output/weights hashes, signature, and safety limits.
- Added nonzero `Hash256` and `Signature64` wrappers so empty model metadata is
  rejected before a model can be admitted; this is explicitly a metadata
  presence gate, not cryptographic verification.
- Added redacted `Debug` output for manifest hashes, signatures, validated
  manifests, and model identity.
- Added scheduler-domain manifest gates: required fallback, bounded evaluation
  steps, bounded memory, bounded confidence, and required telemetry fields.
- Added `PolicyDecision::from_model` so model-backed decisions enforce
  `ModelSafetyLimits::max_confidence` by switching to deterministic fallback
  with `DecisionReason::SafetyRejected` when the model exceeds its declared
  ceiling.
- Added fixed-point scheduler feature validation with `FIXED_POINT_SCALE` across
  every non-priority scheduler input.
- Documented `ScheduleFeatures::priority` as a discrete rank rather than a
  fixed-point ratio.
- Added `VirtAddr::new_x86_64_canonical` as a shared ABI helper for future
  syscall/IPC boundary validation.
- Added a kernel AI-policy smoke that accepts a safe scheduler manifest, rejects
  a manifest without required fallback, and proves deterministic local fallback.
- Updated QEMU image names and manifest metadata to `v0.32.0`.

## Security And Correctness

- AI policy remains advisory only. The kernel enforces deterministic policy and
  capabilities; AI cannot grant authority.
- Model manifests are data, not executable code. Validation returns an opaque
  `ValidatedModelManifest` rather than trusting raw metadata directly.
- `ValidatedModelManifest` now documents that it means structural and policy
  validation only. It does not imply cryptographic verification of model hashes
  or signatures.
- Only fixed-point heuristic/table manifests are accepted in v0.32.0.
  `WasmComponent` and `NeuralNetwork` kinds are declared for the roadmap but
  rejected until real backends and sandboxes exist.
- Scheduler manifests must match the scheduler domain and include the minimum
  scheduler telemetry feature set.
- Deterministic fallback decisions have no model identity, zero confidence, and
  `DecisionReason::DeterministicFallback`.
- The QEMU smoke emits only bounded booleans, schema, local core zero, and
  fallback confidence zero before `[TEST] ai-policy=ok`.
- Actual model loading, inference runtimes, online learning, authority-bearing
  AI decisions, and model-object persistence remain future work.
- The QEMU smoke marker is `manifest_metadata_gate_ok=true`, deliberately not a
  cryptographic safety marker.

## Verification

- `cargo test -p aesynx-ai-policy`.
- `cargo test -p aesynx-kernel ai_policy_smoke`.
- `cargo test -p xtask`.
- `cargo xtask qemu`.
- `cargo fmt --all --check`.
- `scripts/checks.sh`.
- `cargo xtask qemu-suite`.

## Known Limitations

- No AI model is loaded or evaluated.
- No model object is stored in the object registry yet.
- Signature verification is represented by nonzero manifest metadata only; real
  cryptographic verification belongs to future package/model loading work.
- The scheduler still uses deterministic round-robin. v0.33.0 owns the
  non-AI scheduler policy model prototype.

## Release Gate

Before tagging `v0.32.0`:

- Run the normal repository gates.
- Run `cargo xtask qemu-suite`.
- Generate/update SBOM evidence.
- Complete the pentest handoff and store final evidence in
  `security/pentest/v0.32.0.md`.
- Run `cargo xtask release-ready v0.32.0` after the pentest report, SBOM, and
  exact tag commit are ready.

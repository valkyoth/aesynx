# Aesynx v0.19.0 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest and GitHub
checks are green.

## Scope

`v0.19.0` graduates the capability model crate as the authority model that later
kernel capability tables, object access, IPC, drivers, and userspace host calls
will build on. This release keeps capability behavior in safe `no_std` host
tests and does not install a live kernel capability table yet.

## Delivered So Far

- `aesynx-cap` remains dependency-light, `no_std`, and `#![deny(unsafe_code)]`.
- Capability values keep authority-bearing fields private and are neither
  `Copy` nor `Clone`.
- Added a checked `CapId` layout over the ABI `CapId(u64)`: low 32 bits are the
  slot index, high 32 bits are the nonzero capability generation.
- Added `CapSlotIndex`, `CapGeneration`, `CapIdParts`, and `CapIdError`.
- Added `Capability::id_for_slot()` so a capability can derive an external
  table identifier from its generation without exposing internals.
- Permission bitsets continue to reject undefined bits.
- Derive and grant paths continue to reject permission escalation, range
  expansion, stale generations, revoked epochs, and unaudited success.
- Cross-owner derivation still requires `GRANT` and strips re-grant authority
  from the child.
- Revoke authority checks remain explicit and require `REVOKE` on the target
  object before an epoch store may increment.
- Capability `Debug` output is now redacted so object ids, owners, and virtual
  ranges are not dumped accidentally.
- QEMU image names and manifests now use the `v0.19.0` release-candidate
  version. The boot markers are otherwise unchanged from v0.18.1.
- Limine memmap pointer-array bases are validated as aligned high-half
  canonical handoff pointers before the first entry-pointer read.
- Interrupt-gate installation now explicitly rejects callers that arrive with
  IF set after restoring the caller's interrupt state.

## Non-Claims

- No kernel capability table exists yet.
- No `[TEST] cap=ok` serial marker exists yet; that is v0.20.0 scope.
- No syscall, IPC, object, memory-map, driver, package, or userspace operation
  is protected by runtime capabilities yet.
- `CapId` is an index plus generation identifier, not authority by itself.
- No persistent capability store, per-principal table, revoke broadcast, or
  cross-core stale-cache invalidation exists yet.
- The model does not yet bind capability ids to entropy-backed random tokens.
- Future entropy-backed token or capability-id generation must use
  `require_random_tokens()` as the policy gate. Direct `primary_source` reads are
  telemetry/classification only until a grep lint or equivalent test enforces
  the call pattern.

# Aesynx v0.21.0 Release Candidate Notes

Status: release candidate; pending pentest follow-up, GitHub checks, and
release-readiness validation before tag.

## Scope

`v0.21.0` makes capabilities affect a real kernel memory API path. The
capability table from `v0.20.0` now feeds memory-map authorization before the
kernel smoke constructs a mapper-facing checked mapping descriptor.

The scope is still bounded and QEMU-focused: it proves memory capability checks
around the current mapping-descriptor API, not a userspace syscall boundary or
production dynamic memory service.

## Delivered So Far

- Added `MemoryAccess` and `MemoryMapRequest` to `aesynx-cap`.
- Added `Capability::authorize_memory_map` for memory mapping requests.
- Memory authorization requires:
  - `CapKind::Memory`.
  - A requested range contained by the capability range.
  - `MAP` permission.
  - READ permission for every mapping request.
  - WRITE permission for read/write mappings.
  - EXECUTE permission for read/execute mappings.
- Empty and overflowing memory-map requests are rejected before authorization.
- Derived memory subrange capabilities cannot authorize mappings outside their
  delegated range.
- Kernel capability smoke now derives a `MAP|READ` memory subrange cap, checks
  it, and uses it to authorize one read-only checked mapping descriptor.
- Kernel capability smoke verifies missing READ denial, missing WRITE denial,
  and subrange escape denial before reporting success.
- Normal QEMU boot now requires `memory-cap map_allowed=` and
  `[TEST] memory-cap=ok` in addition to the existing `[TEST] cap=ok`.
- QEMU forbidden markers now include `[TEST] memory-cap=fail`.
- Image names, manifest metadata, xtask marker tests, README status, build docs,
  release index, and security controls now track the `v0.21.0` memory
  capability candidate.
- Roadmap docs now explicitly track the future Aesynx SDK path: `aesynx-abi`,
  `aesynx-rt`, native Rust targets, WASM component targets, app templates,
  package/app manifests, and capability-first developer workflows.

## Non-Claims

- No syscall ABI exists yet, so this does not protect ring-3 callers.
- Live dynamic mapper mutation and remapping services remain future work.
- Memory capabilities are not persistent, per-task, SMP-safe, or backed by an
  object registry yet.
- The kernel smoke checks a single-page mapping path. Contiguous mapping,
  protection changes, unmapping, DMA, device memory, and shared-memory services
  still need explicit capability gates in later milestones.
- `CapId` is still an index plus generation identifier. It is not authority by
  itself and is not an entropy-backed bearer token.
- Future entropy-backed token or capability-id generation must use
  `require_random_tokens()` as the policy gate. Direct `primary_source` reads are
  telemetry/classification only until a lint or equivalent test enforces the
  call pattern.
- No authenticated IPC call path, object registry, driver service, package
  service, or native userspace operation is protected by runtime capabilities
  yet.

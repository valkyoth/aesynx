# Aesynx v0.29.0 Release Candidate Notes

`v0.29.0` adds the first scheduler telemetry baseline on top of the v0.28
cooperative executor. The goal is to make local scheduling decisions observable
as bounded, deterministic facts without turning telemetry into authority.

## What Changed

- Added `aesynx-telemetry::SchedulerTelemetry`, a fixed-capacity no_std decision
  trace buffer with checked capacity and typed `CoreId`/`TaskId` fields.
- Added `SchedulerDecisionRecord` and `SchedulerDecisionReason`; the first live
  reason is `RoundRobinRunnable`.
- Scheduler decision `Debug` output redacts selected task identity while keeping
  sequence, core, reason, and bounded queue-depth context visible.
- Queue-depth context carries explicit saturation flags, so future queues larger
  than the redaction bound cannot silently flatten pressure into the maximum
  exported value.
- Extended `TaskTelemetry` with an append-only scheduled-run counter.
- Added `LocalExecutor::dispatch_next_with_telemetry()`, which preflights
  current-task state, queue emptiness, and telemetry capacity before dispatching
  so a full telemetry buffer does not mutate executor state.
- Added a kernel scheduler telemetry smoke that records an A/B/A/B round-robin
  trace, updates per-task scheduled-run counters, samples core run-queue
  telemetry, and reports `[TEST] scheduler-telemetry=ok`.
- Updated QEMU image names/manifests to `v0.29.0` and made the boot smoke
  require `scheduler-telemetry decisions=` plus `[TEST] scheduler-telemetry=ok`.

## Security And Correctness

- Decision records are bounded by compile-time capacity and reject zero-capacity
  or full buffers without wrapping.
- Telemetry-aware dispatch checks trace-buffer capacity before taking a task out
  of the run queue, preserving executor state if telemetry cannot accept the
  record.
- Scheduler decision records keep task IDs available through typed accessors for
  kernel code, but redact them in debug output.
- Core IDs remain visible by design because early telemetry treats core index as
  non-secret local scheduling context; this must be revisited before SMP
  tenancy or cross-tenant scheduling isolation.
- Core telemetry snapshots remain advisory per-counter samples, not coherent
  multi-counter transactions.
- The new scheduler trace is local and cooperative only. It is not a preemption,
  timeslice, cross-core, or userspace scheduling trace.

## Known Limitations

- There is still no architectural context switch, saved register frame,
  preemption, timeslice accounting, syscall ABI, userspace task object, or SMP
  scheduler.
- Scheduler telemetry is an in-memory local trace buffer. It is not yet exported
  through a stable event schema, IPC service, object graph, or userspace world
  query surface.
- Decision reasons are intentionally narrow; priority, deadline, affinity,
  locality, capability pressure, and AI-advisory reasons remain future work.

# Aesynx v0.2.0 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest and GitHub checks
are complete.

## Scope

`v0.2.0` is the toolchain and build-skeleton milestone. It does not boot, create
a disk image, run QEMU, or claim a complete kernel binary pipeline.

This milestone makes the x86_64 kernel build shape reviewable:

- Custom target metadata lives at `targets/x86_64-unknown-aesynx.json`.
- Kernel linker script lives at `linker/kernel-x86_64.ld`.
- Cargo config records the Aesynx target linker and panic policy.
- `cargo xtask build-kernel` validates the target metadata, linker script, Cargo
  config, and `aesynx-kernel` host crate.
- `cargo xtask build-kernel --custom-target-probe` is available for explicit
  nightly `build-std` experiments.
- `cargo xtask image` and `cargo xtask qemu` remain controlled failures until
  `v0.3.0`.

## Pentest Follow-Up Hardening

The v0.2 candidate also tightens skeleton security contracts found during the
temporary pentest handoff:

- Capabilities now have private fields and are not `Copy` or `Clone`.
- Capability derivation and grant paths are production-accessible only through
  audited APIs.
- Revocation checks borrow authority capabilities instead of duplicating them.
- IPC separates caller-provided `MessageRequest` from kernel-stamped
  `MessageHeader`.
- Inline IPC payloads expose a length-bounded `as_slice()` accessor.
- AI policy confidence uses a bounded `Confidence` newtype.
- Architecture memory activation, TLB flush, and clock reads are fallible APIs.
- Scheduler priority and task budget values are bounded by constructors.
- Core telemetry has explicit update methods, and task telemetry is marked
  single-writer/non-`Sync`.
- Bytecode branch targets and memory offsets use explicit `Unchecked*` wrapper
  types to force future executor validation.
- Secret scanning now includes Markdown files.

## Stable Toolchain Position

Aesynx remains on stable Rust `1.96.0`. Stable Cargo still rejects the
nightly-only `-Z build-std=core` path required for ordinary custom JSON target
kernel-object experiments. The stable release gate therefore validates the
build shape and host kernel crate without silently depending on nightly.

Nightly use is allowed only through the explicit probe command and is not part
of the `v0.2.0` release gate.

## Verification

Required before pentest handoff:

```bash
cargo fmt --all --check
cargo check --workspace
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cargo deny check
cargo audit
scripts/checks.sh
scripts/generate-sbom.sh
cargo xtask build-kernel
```

Expected controlled failures:

```bash
cargo xtask image
cargo xtask qemu
```

Both commands must return the documented "not implemented until v0.3.0" result
until the boot image pipeline lands.

## Known Limitations

- No bootloader integration yet.
- No kernel `_start` entry yet.
- No QEMU image yet.
- No serial boot marker yet.
- Custom target object builds require a future stable path or an explicitly
  documented nightly exception.

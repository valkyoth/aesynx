# Aesynx v0.7.0 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest and GitHub checks
are complete.

## Scope

`v0.7.0` is the first x86_64 CPU descriptor-table milestone. It keeps the v0.6
early diagnostics and Limine BootInfo path, then installs an Aesynx-owned GDT,
TSS, and dedicated double-fault stack during early `_start`.

It does not claim IDT installation, exception handling, page-fault decoding,
interrupt delivery, SMP TSS setup, privilege transitions, syscall/sysret setup,
or double-fault handler execution. Those begin in later milestones.

## Deliverables

- `aesynx-arch-x86_64::descriptors` module.
- Safe public descriptor setup API.
- Private static GDT with null, kernel code, kernel data, and TSS descriptors.
- CS reloaded, SS/DS/ES reloaded to the kernel data selector, and FS/GS reset
  to null selectors after `lgdt` so Aesynx is not running on stale bootloader
  descriptors.
- Private static x86_64 TSS.
- Dedicated aligned 16 KiB double-fault stack registered in IST slot 1.
- Unsafe future-facing `set_ring0_stack()` API documents the required RSP0
  setup before any ring 3 to ring 0 transition is allowed.
- Early kernel boot phase `cpu-setup`.
- Structured diagnostic record for GDT/TSS initialization.
- Serial summary line for GDT entries, TSS selector, double-fault IST, and stack
  size.
- `[TEST] gdt=ok` serial marker.
- v0.7 QEMU image, manifest, and serial-log names.
- QEMU boot and panic-smoke gates require the GDT/TSS marker.

## Security Notes

- Descriptor setup is isolated to the architecture crate.
- The public descriptor initialization API is safe; unsafe assembly and static
  descriptor mutation stay private to `aesynx-arch-x86_64::descriptors`.
- Descriptor tables are initialized once during early single-core boot before
  Aesynx enables interrupts.
- GDT installation now reloads the live code and data segment registers,
  resets FS/GS selectors to null for future TLS/per-CPU setup, and then loads
  TR, avoiding stale Limine segment caches.
- The double-fault stack is private, aligned, and reserved for future IDT use.
- v0.7 does not yet install an IDT, so the double-fault stack is registered in
  the TSS but not exercised by an exception handler.
- TSS.rsp0 is intentionally not active yet; future userspace bringup must call
  `set_ring0_stack()` with a valid per-core kernel stack before enabling ring 3.
- QEMU normal and panic-smoke paths both verify `[TEST] gdt=ok`.


## Known Limitations

- No IDT exists yet.
- No exception handler uses the double-fault IST yet.
- No per-core TSS exists yet.
- No user/kernel privilege transition uses the GDT yet.
- No RSP0 stack is installed yet because ring 3 execution is not enabled.
- No interrupt-state control is claimed.

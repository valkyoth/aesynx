# Aesynx v0.13.0 Release Candidate Notes

Status: implementation candidate; pending pentest and GitHub checks before tag.

## Scope

`v0.13.0` starts the physical and virtual memory phase by turning the normalized
boot memory map into checked accounting that later allocator work can trust. The
release reports total, usable, reserved, kernel, bootloader, framebuffer, ACPI,
bad-memory, and full-frame counts during normal QEMU boot.

This is not a frame allocator yet. It does not claim page-table ownership,
bootloader memory reclamation, heap allocation, page-fault recovery, or APIC
MMIO mapping.

## Deliverables

- Checked `aesynx-boot::MemoryMap::summary()` accounting.
- BootInfo normalization rejects overlapping memory regions before accounting.
- BootInfo normalization rejects memory maps whose aggregate accounting
  overflows.
- `MemoryAccountingError::Overflow` for fail-closed memory-map totals.
- Total, usable, and reserved byte counts.
- Full-frame accounting using 4 KiB frame boundaries.
- Kernel, bootloader, framebuffer, ACPI, and bad-memory reserved byte totals.
- Kernel boot summary propagation of the checked memory report.
- Normal QEMU boot serial markers for total, usable, reserved, and
  `[TEST] memory-map=ok`.
- v0.13 QEMU image, manifest, serial-log names, help text, and marker tests.

Expected normal boot serial shape:

```text
memory total_bytes=<n> total_frames=<n> regions=<n>
memory usable_bytes=<n> usable_frames=<n> usable_regions=<n>
memory reserved_bytes=<n> reserved_frames=<n> reserved_regions=<n> kernel_bytes=<n> bootloader_bytes=<n> framebuffer_bytes=<n> acpi_bytes=<n> bad_bytes=<n>
[TEST] memory-map=ok
[TEST] bootinfo=ok
[TEST] boot=ok
```

## Security Notes

- Memory region end addresses, overlapping ranges, and aggregate accounting are
  validated during BootInfo normalization.
- Summary totals use checked arithmetic and return `MemoryAccountingError`
  instead of saturating or wrapping.
- Frame counts only include full 4 KiB frames contained inside each region.
- Non-usable memory contributes to reserved accounting; kernel and bootloader
  regions are tracked separately inside that reserved total.
- If memory accounting fails during normal boot, the kernel emits
  `[TEST] memory-map=fail` and does not emit boot success markers.
- Memory-region debug output continues to redact physical starts.

## Known Limitations

- No frame allocator exists yet.
- No page-table mapper exists yet.
- No heap allocator exists yet.
- No bootloader memory reclamation exists yet.
- No ownership transition for usable frames exists yet.
- No page-fault recovery exists yet.

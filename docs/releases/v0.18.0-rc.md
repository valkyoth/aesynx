# Aesynx v0.18.0 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest and GitHub
checks are green.

## Scope

`v0.18.0` replaces the v0.17 bump-only heap with a bounded reusable kernel heap.
The release keeps the heap static and small, but adds fixed slab classes,
page-sized runs for larger allocations, aggregate stats, checked deallocation,
and QEMU stress evidence before later capability and object structures depend
on long-lived allocations.

## Delivered So Far

- `crates/aesynx-kernel/src/kernel_heap/` now owns the kernel allocator in
  split modules for layout constants, allocator logic, status/error types, and
  host tests.
- The global allocator is now `KernelHeapAllocator` instead of the removed
  bump-only `EarlyBumpAllocator`.
- The retained linker section is now `.aesynx_kernel_heap`.
- The heap is 256 KiB, page-aligned, initialized once after Aesynx-owned CR3
  activation and post-CR3 CPU hardening.
- Fixed slab classes cover 16, 32, 64, 128, 256, 512, 1024, and 2048 byte
  allocations.
- Larger allocations use page-sized runs inside the static kernel heap.
- Checked deallocation detects invalid frees and double frees; `GlobalAlloc`
  still drops deallocation errors because the Rust allocator ABI cannot return
  them.
- Invalid-free attempts are now surfaced in heap stats and the QEMU smoke
  contract as `invalid_free_detected=true`.
- Slab pages track live allocations with per-page counters, so normal frees no
  longer need a full free-list walk to decide whether the page is empty.
- Freed slab blocks and page-sized runs are zeroed before reuse to reduce stale
  kernel data remanence.
- A delayed stale raw-pointer free after the same address has been reallocated
  remains outside what the standard `GlobalAlloc::dealloc(ptr, layout)` ABI can
  distinguish; future ownership-token or quarantine work must close that class.
- QEMU normal boot still smokes `Box`, `Vec`, and `BTreeMap` allocations.
- The heap smoke now also proves slab reuse, large page-run allocation/free,
  mixed allocation/free stress, invalid-free telemetry, double-free detection,
  and oversized allocation rejection.
- Host tests cover pre-initialization rejection, one-shot initialization, slab
  reuse, page-run reuse, invalid-free telemetry, double-free detection, zeroing
  before reuse, stats accounting, and OOM without stat advancement.
- QEMU image names and manifests now use the `v0.18.0` release-candidate
  version.
- Xtask tests now pin the v0.18.0 boot, panic, exception, and timer artifact
  names plus the manifest release title.

Expected heap status shape:

```text
heap bytes=<n> allocated=<n> peak=<n> slab_classes=<n> slab_allocations=<n> page_allocations=<n> frees=<n> double_free_detected=true invalid_free_detected=true box_ok=true vec_ok=true btree_ok=true slab_reuse_ok=true page_run_ok=true stress_ok=true oom_rejected=true
[TEST] heap=ok
```

- `cargo test -p aesynx-kernel kernel_heap`.
- `cargo xtask qemu`.
- `cargo xtask qemu-suite`.
- `scripts/checks.sh`.

## Non-Claims

- Page-backed allocation means page-sized runs inside the static kernel heap,
  not dynamic growth from the physical frame allocator.
- The allocator still uses one global metadata lock; per-core heaps and
  remote-free message passing remain future SMP work.
- No quarantine, allocation backtraces, or full leak report exists yet.
- The standard global allocator deallocation ABI cannot prove that a raw pointer
  belongs to the current allocation epoch after that address has been reused.
- No allocation-while-locking policy exists yet.
- No user allocator, process heap, or userspace ABI.

# Aesynx v0.4.0 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest and GitHub checks
are complete.

## Scope

`v0.4.0` is the first Rust kernel serial-boot milestone. It replaces the v0.3
stage-0 probe with a freestanding Rust kernel ELF loaded by Limine.

It does not claim BootInfo parsing, memory-map normalization, page-table
ownership, interrupts, allocator setup, or production panic diagnostics. Those
start in later milestones.

## Deliverables

- `x86_64-unknown-none` stable boot target in `rust-toolchain.toml`.
- `boot/qemu/limine.conf`.
- Rust `_start` in `crates/aesynx-kernel/src/main.rs`.
- Minimal panic handler.
- x86_64 COM1 serial writer.
- Reviewed unsafe boundary for x86_64 port I/O.
- Release-profile Limine ISO generation in `cargo xtask image`.
- QEMU serial marker validation in `cargo xtask qemu`.

Expected serial:

```text
Aesynx: booting
arch=x86_64 platform=qemu
[TEST] boot=ok
```

## Security Notes

- The default crate rule remains `#![no_std]` and `#![deny(unsafe_code)]`.
- The first implementation unsafe boundary is isolated to x86_64 port I/O and
  documented in `docs/unsafe-policy.md`.
- The COM1 writer bounds transmit polling so panic output cannot spin forever
  on a missing or wedged UART.
- The early COM1 type is marked single-core/non-`Sync`; real SMP-safe logging
  requires a future logger-core or lock design.
- x86_64 port I/O uses a typed admitted-port set instead of accepting arbitrary
  `u16` port addresses.
- The Rust 2024 `unsafe(no_mangle)` entry symbol is documented as an entry ABI
  boundary.
- The linker script page-aligns executable, read-only, and writable load
  segments so Limine does not load mixed-permission program headers onto the
  same page.
- The boot entry writes fixed strings directly; the v0.4 boot ELF has no `.got`
  section.
- Runtime environment capability IDs are private and read through accessors.
- `cargo xtask image` rejects Limine versions older than 12.3.2 and records
  the Rust, Limine, xorriso, and QEMU version banners in the image manifest.
- GitHub CI installs a checksum-pinned Limine 12.3.2 source release and runs
  the QEMU serial boot smoke.
- `boot/qemu/limine.conf` leaves KASLR off for the v0.4 QEMU smoke image until
  Limine BootInfo parsing can populate `KernelImageInfo` with randomized load
  addresses.
- v0.4 is not yet a reproducible release-artifact pipeline.


## Known Limitations

- BootInfo is not consumed yet.
- Limine handoff metadata is not normalized yet.
- KASLR is disabled in the QEMU smoke config until BootInfo normalization lands.
- The serial writer is an early COM1 diagnostic path, not a full driver.
- Serial output is single-core-only for v0.4; SMP logging needs a later
  synchronization design.
- `X86_64::halt_forever()` still uses a spin loop rather than `hlt`.
- No page-table, interrupt, allocator, or panic-reporting subsystem is active.

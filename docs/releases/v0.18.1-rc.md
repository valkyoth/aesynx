# Aesynx v0.18.1 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest and GitHub
checks are green.

## Scope

`v0.18.1` makes early entropy and identifier-generation semantics explicit
before capability and object identifiers become security-relevant. The release
does not create cryptographic tokens yet; it separates monotonic
anti-confusion generations from attacker-unpredictable random-token policy and
records whether x86_64 hardware entropy features are present.

## Delivered So Far

- Added the dependency-free `aesynx-entropy` crate.
- Added explicit entropy source labels for `RDSEED`, `RDRAND`, and deterministic
  monotonic fallback.
- Added source quality labels that distinguish hardware seed/random material
  from deterministic anti-confusion fallback.
- Added `GenerationCounter`, which fails instead of wrapping.
- Added random-token policy gating so deterministic fallback cannot satisfy
  attacker-unpredictable token requirements.
- Added x86_64 CPUID detection for `RDRAND` and `RDSEED` support.
- Added a normal-boot entropy policy smoke after CPU hardening and before heap
  initialization.
- Serial output reports only booleans and the selected source label; no random
  material is read or logged in this release.
- QEMU image names and manifests now use the `v0.18.1` release-candidate
  version.
- Xtask tests now pin the v0.18.1 boot, panic, exception, and timer artifact
  names plus the manifest release title and entropy markers.

Expected entropy status shape:

```text
entropy-policy rdrand=<bool> rdseed=<bool> hardware_present=<bool> fallback_used=<bool> generation_counter_ok=true random_tokens_available=<bool> source=<source>
[TEST] entropy-policy=ok
```

## Non-Claims

- `RDRAND` and `RDSEED` are classified by CPUID only; this release does not
  consume, mix, whiten, or persist hardware random material.
- Hardware entropy is treated as one future input, not as a sole trust anchor.
- Deterministic fallback is suitable only for monotonic anti-confusion
  generation counters, not for attacker-unpredictable tokens.
- No CSPRNG, entropy pool, DRBG, virtio-rng, TPM, bootloader seed, or persistent
  seed exists yet.
- No capability, object, package, or boot signature format depends on these
  identifiers yet.

# Aesynx v0.16.3 Release Candidate Notes

Status: tagged; pentest follow-up, GitHub checks, and release-readiness
validation passed before tag.

## Scope

`v0.16.3` starts the hardware hardening step after `v0.16.2` proved Aesynx can
survive a terminal CR3 switch to an Aesynx-owned root table. The first slice
adds CPUID-gated x86_64 hardening for NX, write-protect, SMEP, SMAP, and UMIP,
with redacted QEMU status output.

## Delivered So Far

- `aesynx-arch_x86_64::cpu_hardening` derives a host-testable hardening plan
  from CPUID capabilities.
- NX is required for this release path. If CPUID does not report NX support,
  the hardening plan fails closed before EFER.NXE is written.
- EFER.NXE is enabled after CPUID reports NX support.
- CR0.WP is enabled so supervisor writes respect read-only page permissions.
- CR4.SMEP, CR4.SMAP, and CR4.UMIP are enabled only when CPUID reports each
  feature.
- Normal QEMU boot enables CPU hardening after the Aesynx-owned CR3 root is
  loaded.
- QEMU boot smoke now requires `cpu-hardening nx=<bool> wp=<bool>
  smep=<bool> smap=<bool> umip=<bool>` and `[TEST] cpu-hardening=ok`.
- Hardening status output reports read-back booleans only, verifies requested
  bits stuck, and does not dump raw EFER, CR0, or CR4 values.
- The activation stack now lives in a dedicated page-aligned linker section
  outside the normal data/BSS policy range.
- The linker reserves one unmapped 4 KiB guard page immediately before the
  activation stack.
- Kernel mapping smoke explicitly maps the activation stack pages, verifies
  RW/NX/kernel/local/normal-memory flags, and verifies the guard page remains
  unmapped before CR3 activation.
- QEMU boot smoke now requires `kernel_stack_pages=<n>`,
  `kernel_stack_guard_ok=true`, and `[TEST] kernel-stack-guard=ok`.

## Pentest Follow-Up

- `Capability::derive_*` no longer permits cross-principal ownership transfer
  with DERIVE alone; cross-owner derivation now also requires GRANT.
- Unbounded parent capabilities now reject child ranges whose `base + len`
  overflows.
- Added live-generation/epoch checked `derive_live_with_audit` and
  `grant_live_with_audit` paths that reject stale or revoked parents before
  audit emission.
- `set_ring0_stack` now validates null, lower-half, noncanonical, and
  non-16-byte-aligned RSP0 values in release builds before writing TSS.rsp0.
- Limine framebuffer, HHDM, and RSDP payload addresses are checked as
  high-half canonical handoff addresses before entering BootInfo.
- Limine response-pointer validation now accepts an explicit architecture VMA
  floor instead of embedding the x86_64 floor inside the generic helper.
- `BootInfo` kernel-image validation now checks both virtual start and virtual
  end against the architecture kernel VMA floor.
- SMP descriptor-table risk is tracked as a blocker: the current single-core
  static GDT/TSS/IDT storage and non-atomic IDT gate writes must be replaced or
  guarded before secondary cores can rely on shared descriptor state.
- Cross-owner derived capabilities now strip GRANT from the child, matching
  one-hop grant behavior.
- CPU hardening status now reports read-back EFER/CR0/CR4 state and fails if a
  requested hardening bit does not stick.
- The bytecode verifier now exposes register and endpoint bound checks before
  an interpreter exists, matching the existing offset and branch-target gates.
- Static activation arena/stack migration to explicit interior mutability or
  per-core ownership is tracked as an SMP blocker.
- The intentional panic path remains confined to the opt-in `panic-smoke`
  feature and separate QEMU panic image.

## Non-Claims

- No audited SMAP usercopy access-window helpers yet.
- No per-core hardening policy or SMP bring-up yet.
- No PCID/INVPCID or TLB shootdown model yet.
- No userspace transition or process isolation yet.
- The stack guard currently covers the terminal activation stack only. General
  per-task, per-core, interrupt, and scheduler stacks remain future work.

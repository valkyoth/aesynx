# Aesynx v0.26.0 Release Candidate Notes

Status: release candidate

## Summary

`v0.26.0` moves service queues from the host-side model into the kernel-facing
`aesynx-ipc` crate. The kernel now has fixed-capacity no_std queues for log,
timer, and object services, plus a QEMU smoke path that submits and observes a
log service request through the queue before reporting `[TEST] service-queue=ok`.

## Added

- Added typed live-core validation primitives to `aesynx-ipc`:
  `LiveCoreSet`, `ValidatedCoreId`, and `CoreValidationError`.
- Changed kernel message stamping so `MessageHeader::stamp` requires validated
  source and destination core IDs.
- Added `ServiceKind`, `ServiceRequest`, `ServiceCompletion`, completion
  statuses, and request validation to the no_std IPC crate.
- Added a fixed-capacity no_std `ServiceRingQueue` with release/acquire
  ordering evidence.
- Added `ServiceQueuePair` and `ServiceQueueSet` for log, timer, and object
  service queues.
- Added a kernel service-queue smoke that submits log, timer, and object
  requests through the queue set, observes the log request, posts a completion,
  and verifies unsupported service routing fails closed.
- Added QEMU marker coverage for `service-queue log_submitted=` and
  `[TEST] service-queue=ok`.

## Security And Correctness

- Raw `CoreId` values are not accepted as queue-routing authority for message
  headers; destinations must pass through a typed live-core-set check first.
- Zero message IDs are rejected for kernel service requests.
- Zero-capacity service queues are rejected.
- Full queue pushes fail before mutating queue state.
- Empty queue pops fail before mutating queue state.
- FIFO ordering is preserved across ring wraparound.
- Queue publication/observation records the intended release/acquire contract.
- Service request and completion debug output redacts message IDs, caller core
  IDs, payloads, capabilities, and object IDs.
- The service queue set accepts only the v0.26 log, timer, and object service
  skeletons; unsupported services fail closed without mutating supported queues.

- `cargo xtask qemu-suite`.

## Known Limitations

- Service queues are still kernel-internal smoke infrastructure. There is no
  userspace syscall ABI, scheduler wakeup path, blocking receive, interrupt
  doorbell, SMP producer/consumer queue, or capability-mediated endpoint object
  binding yet.
- The log, timer, and object services are skeleton queues. The smoke validates
  queue routing and completion mechanics, not real service-side work.
- Ordering evidence records the intended release/acquire contract. The current
  single-core smoke does not yet exercise shared-memory multi-core contention.

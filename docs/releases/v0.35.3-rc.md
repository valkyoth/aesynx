# Aesynx v0.35.3 Release Candidate Notes

`v0.35.3` adds a formal AP startup state table. It does not start secondary
cores yet; it makes the joint `CoreHardwareState`, `CoreAssignmentState`, and
`CoreState` model explicit before real AP execution can consume it.

## What Changed

- Added `CoreStartupJointState` to `aesynx-core`.
- Added an audit helper that counts the full startup-state cross-product.
- Valid startup combinations are now explicit:
  discovered/offline, startup-staged/booting, online/online, and
  quarantined/quarantined, each with assigned or unassigned role state.
- `CoreTopology` now validates startup staging, role assignment,
  hardware-online marking, and quarantine through the shared table before
  mutation.
- Topology mutation revalidates the resulting joint state before committing the
  updated entry.
- Added host tests for valid and invalid joint states, transition intent
  helpers, and table cardinality.
- Added QEMU smoke evidence for `state_table_ok=true`.
- Renamed the AI manifest admission result to
  `StructurallyCheckedModelManifest` so structural policy checks cannot be
  confused with cryptographic hash/signature verification.
- Renamed the private AI metadata comparison helper to make clear it is
  structural equality only, not an authentication primitive.
- Increased AP startup preflight stack minimums from 16 KiB to 32 KiB.
- Made boot-barrier arrival accounting and capability revoke generation
  increments use checked arithmetic.
- Hardened generated image manifest sanitization so dynamic path/tool-version
  fields reject ASCII control characters and `=` before evidence is written.
- Changed `aesynx-sync::EarlyLock::try_lock_irq()` to require an explicit
  `ArchIrqDisableProof` argument, with the current constructor deliberately
  marked as model-only for single-core smoke/tests.
- Moved current image and manifest names to `v0.35.3`.
- Re-scoped the release plan so v0.35.3 is the AP state-table release and real
  x86_64 AP startup is tracked after the v0.35.4 hardening blockers.

## Security Notes

- This closes the design gap where hardware state, assignment state, and local
  core state could drift as three related but separate axes.
- State transitions remain owner-scoped and fail before mutation on invalid
  joint states.
- The QEMU smoke now proves every modeled four-core entry remains in a valid
  joint state after startup evidence, preflight, hardware-online marking, and
  boot-barrier setup.
- Shared bootstrap-only descriptors remain an explicit execution blocker. QEMU
  still requires `ap_execution_blocked_ok=true`.
- Future APIC INIT/SIPI dispatch must consume a sealed dispatch token derived
  from an execution-allowed preflight; this remains planned for the real AP
  startup milestone and is not claimed by v0.35.3.
- The current `smp` compile-time tripwire is documented as a release blocker,
  not a runtime protection for the shared IDT/static descriptor-table design;
  live IDT mutation still requires per-core tables, shadow-IDT replacement, or
  NMI-source quiescing before runtime use.
- The IRQ-lock proof token prevents silently treating `LocalInterruptMask` as
  hardware enforcement, but the active proof remains model-only until the
  architecture crate provides a real interrupt-disable token.
- No attacker-unpredictable token consumer was added; QEMU still reports
  `drbg_self_test=false`, and random-token use remains blocked until the DRBG
  implementation exists.

## QEMU Evidence

Expected normal boot evidence includes:

```text
multicore-topology qemu_smp_cores_ok=true hardware_online_ok=true role_assignment_ok=true state_table_ok=true bootstrap_ok=true scheduler_ok=true driver_service_ok=true idle_ok=true startup_evidence_ok=true ap_preflight_ok=true ap_execution_blocked_ok=true multicore_barrier_ok=true
[TEST] multicore-topology=ok
```

The generated manifest name is:

```text
name=Aesynx v0.35.3 AP state-table candidate
```

## Non-Claims

- No AP startup trampoline.
- No secondary core executes Rust code yet.
- No per-core GDT, IDT, TSS, or IST setup yet.
- No directed IRQ routing to AP-owned service domains yet.
- No cross-core message fabric yet.

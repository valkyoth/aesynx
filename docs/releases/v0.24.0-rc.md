# Aesynx v0.24.0 Release Candidate Notes

Status: release candidate

## Summary

`v0.24.0` adds the no_std kernel-facing local object registry. The registry is
fixed-capacity, safe Rust, and validates object identity, ownership, deletion,
listing, and capability-backed references before the later service-queue
milestones expose object operations through kernel services.

## Added

- Split `aesynx-object` into small `types` and `registry` modules.
- Added `ObjectRecord` with object ID, object type, owner core, and object
  generation.
- Added `ObjectCreate` constructors for memory, endpoint, queue, and
  task-placeholder objects.
- Added `ObjectRegistry<const CAPACITY: usize>` with create, get, list, delete,
  len, capacity, and empty checks.
- Added `resolve_capability()` to verify that a capability references a live
  registry object with matching object ID, object kind, generation, and required
  permission.

## Security And Correctness

- Zero object IDs are rejected.
- Duplicate live IDs are rejected.
- Deleted slots are recycled with a checked next-generation counter, so
  create/delete churn cannot permanently exhaust a fixed-capacity registry while
  stale capabilities against a recreated object ID still fail closed.
- `list()` validates the caller-provided output capacity before writing any
  records.
- Object record `Debug` output redacts object identifiers.
- Capability resolution rejects missing permissions, wrong capability kinds,
  stale object generations, and missing objects.
- Memory map authorization now requires an otherwise-authorized `Memory`
  capability to carry a bounded range. Unbounded memory roots remain usable as
  kernel-internal derivation roots, but cannot directly authorize arbitrary map
  requests.
- Cross-owner capability derivation and grant now strip `GRANT`, `REVOKE`, and
  `ADMIN` from delegated children, so another owner cannot receive revocation
  or administration authority by accident. Same-owner derivation keeps the
  existing explicit authority-retention behavior, while `grant()` still removes
  regrant authority by default.
- The early x86_64 IDT now installs deterministic halt-and-log catch-all stubs
  for every vector before overriding the breakpoint, page-fault, and
  double-fault entries. Unexpected exceptions therefore produce a diagnostic
  halt instead of falling through a missing gate into an uncontrolled triple
  fault.
- `set_ring0_stack()` now rejects release builds with interrupts enabled before
  mutating the current global TSS `rsp0` field. The API remains unsafe because
  future ring-3 work must still prove a valid current-core kernel stack and
  replace global descriptor storage before SMP.
- Kernel heap free-list decoding gained a debug assertion against the empty
  sentinel as a defense-in-depth backstop for guarded call sites.
- CET/shadow-stack support is not implemented in this release and is tracked as
  future CPUID-gated CPU hardening work.

## Known Limitations

- The registry is not yet wired into a kernel service queue, syscall path, or
  persistent object store.
- `CapabilityTable::insert_root()` remains a kernel-internal mint path. Before
  any syscall can trigger root minting, it needs either a `Mint` audit event or
  explicit object-registry lifecycle audit correlation.
- Registry lookup is still a bounded linear scan. This is acceptable for the
  current fixed-capacity host model, but any syscall-reachable object table with
  larger capacity needs an indexed lookup path or strict per-owner quotas before
  it becomes a hot authority check.
- The QEMU serial marker set is unchanged from `v0.23.0`; this release adds
  host-tested no_std registry logic for the next service milestones.
- Page-fault and double-fault stubs still halt by design; recoverable fault
  handling is a later userspace/address-space milestone.
- The slab double-free check still performs a bounded free-list walk under the
  heap lock. This is acceptable for the current fixed heap size and remains a
  future scaling item before larger or per-core heaps.
- Non-atomic IDT descriptor writes remain scoped to early/single-core setup and
  documented as an NMI limitation until per-core/SMP descriptor work replaces
  the current storage model.
- The CR3 overlap check remains scoped to single-core boot and must be revisited
  with SMP and TLB shootdown policy.

# Aesynx v0.24.0 Release Candidate Notes

Status: release candidate

## Summary

`v0.24.0` adds the no_std kernel-facing local object registry. The registry is
fixed-capacity, safe Rust, and validates object identity, ownership, deletion,
listing, and capability-backed references before the later service-queue
milestones expose object operations through kernel services.

## Added

- Split `aesynx-object` into small `types` and `registry` modules.
- Added `ObjectRecord` with object ID, object type, owner core, and object
  generation.
- Added `ObjectCreate` constructors for memory, endpoint, queue, and
  task-placeholder objects.
- Added `ObjectRegistry<const CAPACITY: usize>` with create, get, list, delete,
  len, capacity, and empty checks.
- Added `resolve_capability()` to verify that a capability references a live
  registry object with matching object ID, object kind, generation, and required
  permission.

## Security And Correctness

- Zero object IDs are rejected.
- Duplicate live IDs are rejected.
- Deleted object IDs become tombstones and cannot be reused by the v0.24
  registry, so stale references fail closed.
- `list()` validates the caller-provided output capacity before writing any
  records.
- Object record `Debug` output redacts object identifiers.
- Capability resolution rejects missing permissions, wrong capability kinds,
  stale object generations, missing objects, and deleted objects.

## Verification

- `cargo test -p aesynx-object`.
- `cargo check -p aesynx-object --target x86_64-unknown-none`.
- `cargo fmt --all --check`.
- `scripts/checks.sh`.
- `cargo xtask qemu-suite`.

## Known Limitations

- The registry is not yet wired into a kernel service queue, syscall path, or
  persistent object store.
- Deleted object IDs are retired permanently inside the current registry rather
  than recycled with a persistent generation store.
- The QEMU serial marker set is unchanged from `v0.23.0`; this release adds
  host-tested no_std registry logic for the next service milestones.

## Release Gate

- `cargo xtask release-ready v0.24.0` after the pentest report, SBOM, and exact
  commit evidence are present.

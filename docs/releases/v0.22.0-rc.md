# Aesynx v0.22.0 Release Candidate Notes

Status: implementation candidate; pending pentest follow-up, GitHub checks, and
release-readiness validation before tag.

## Scope

`v0.22.0` makes capability authority movement visible. The capability table now
requires audited table revocation, grant is exercised in the kernel smoke, audit
debug output is redacted, and QEMU records capability-audit and cap-fault
telemetry evidence before the boot smoke can pass.

The scope is still bounded and QEMU-focused: it proves the table/audit model and
serial evidence path for kernel-owned smoke code, not a production per-task
audit log, persistent object history, or authenticated userspace syscall path.

## Delivered So Far

- Added `CapAuditAction::Revoke`.
- Added `CapAuditError` so audit sinks no longer use derivation errors as their
  transport error type.
- Added `RedactedCapAuditEvent` and a manual `Debug` implementation for
  `CapAuditEvent`; debug output exposes only action, permission bits,
  cross-owner status, and affected slot count.
- Audit events now include `affected_slots`.
- `CapabilityTable::revoke_with_audit` validates revoke authority, generation
  overflow, and affected slot count before emitting the revoke audit event and
  committing total table revoke.
- The previous public unaudited table revoke path has been removed from the
  capability-table API.
- Added host coverage for audited grant, audited revoke, affected slot counts,
  and audit debug redaction.
- Added redacted cap-fault telemetry events to `aesynx-telemetry`; events expose
  only fault kind and aggregate count.
- Kernel capability smoke now exercises audited derive, audited grant, audited
  revoke, missing-permission cap faults, stale-id cap faults, and memory
  capability authorization in one post-CR3 path.
- Normal QEMU boot now requires `cap-audit events=` and `[TEST] cap-audit=ok`
  in addition to `[TEST] cap=ok` and `[TEST] memory-cap=ok`.
- QEMU forbidden markers now include `[TEST] cap-audit=fail`.
- Image names, manifest metadata, xtask marker tests, README status, build
  docs, release index, and security controls now track the `v0.22.0` capability
  audit candidate.

## Non-Claims

- No persistent audit log exists yet.
- No authenticated IPC, syscall, object registry, package service, driver
  service, or userspace operation consumes these events yet.
- Audit events are bounded model events, not tamper-proof storage.
- Cap-fault telemetry is aggregate and advisory. It must not become an
  authorization source.
- Permission bits are still visible in redacted audit summaries because they are
  policy metadata; object IDs, principal IDs, generations, and revocation epochs
  remain out of debug output.
- Memory capabilities remain single-kernel-smoke scoped and are not per-task,
  SMP-safe, persistent, or backed by an object registry.

# Aesynx v0.16.2 Release Candidate Notes

Status: implementation candidate; not tag-ready until pentest is green and
GitHub checks are green.

## Scope

`v0.16.2` turns the verified v0.16 mapping policy into a terminal QEMU boot
handoff that survives a CR3 switch to an Aesynx-owned x86_64 root table. The
release exports audited mapper state into hardware-shaped entries, writes them
to a static kernel activation arena, switches to a private kernel activation
stack, loads CR3, and emits post-switch serial evidence.

## Delivered So Far

- `aesynx-mm` can export an audited `PageTableMapper` into an
  `X86_64PageTableImage`.
- Hardware-image export validates the physical root-table arena, rewrites
  model next-table indices into physical next-table entries, and preserves leaf
  entries.
- Intermediate hardware entries derive writable/user bits from their child
  subtree instead of blindly granting every parent entry maximum authority.
- Hardware-image debug output redacts physical addresses and raw table entries.
- Kernel mapping smoke now exports the policy-derived mapper to a hardware
  image and requires `hardware_image_ok=true` before
  `[TEST] paging-policy-model=ok`.
- Kernel mapping smoke now allocates the page-table arena through the checked
  bitmap frame allocator seeded from normalized BootInfo memory regions.
- Kernel mapping smoke now derives text, rodata, and data physical mappings
  from Limine's normalized kernel image placement instead of a placeholder
  physical base.
- QEMU boot smoke now requires `hardware_arena_frames=<n>` and
  `hardware_root_allocated=true`, proving the hardware-image root is
  allocator-issued before CR3 activation work consumes it.
- Kernel mapping smoke now streams used hardware-image tables into a
  page-aligned static kernel activation arena before reporting success. This
  avoids early-boot stack pressure from a full 16 KiB image value and avoids
  relying on HHDM writes before Aesynx owns CR3.
- QEMU boot smoke now requires `hardware_tables_copied=<n>` and
  `hardware_copied=true`, proving the copy path executed.
- `aesynx-arch-x86_64` now exposes a tightly-scoped unsafe CR3 load primitive
  documented in the unsafe policy.
- Normal QEMU boot now switches from the Limine boot stack to a private static
  kernel activation stack before loading the Aesynx-owned CR3 root.
- QEMU boot smoke now requires `kernel-cr3 active=true` and
  `[TEST] kernel-cr3=ok`, proving execution continued after CR3 activation.
- QEMU smoke metadata now records the hardware-copy and kernel-CR3 markers.

## Non-Claims

- The active CR3 replacement is a terminal boot smoke. General post-switch
  kernel services, scheduling, and userspace are still future milestones.
- No full process isolation or userspace enforcement yet.
- No kernel stack guard or CPU hardening-bit enablement yet.
- No PCID/INVPCID, TLB shootdown model, bootloader table reclamation, or Limine
  memory reclamation yet.

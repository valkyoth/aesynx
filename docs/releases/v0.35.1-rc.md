# Aesynx v0.35.1 Release Candidate Notes

`v0.35.1` adds the AP startup evidence contract. It does not start secondary
cores yet; it removes the public topology path that could mark a staged core
online without ticket-derived arrival evidence.

## What Changed

- Added `CoreStartupTicket`, issued only by owner-scoped topology startup
  staging.
- Added `CoreStartupArrival`, derived only from a matching startup ticket.
- Changed hardware-online transition so it validates AP arrival evidence for
  the target core, hardware ID, coordinator core, and exact staged startup
  epoch.
- Reworked the four-core QEMU topology smoke to use startup tickets and arrival
  evidence before recording hardware-online state.
- Changed hardware-online transition to consume `CoreStartupArrival` by value,
  so arrival evidence is single-use at the API boundary.
- Removed the ticket-discarding `CoreTopology::stage_startup` wrapper; callers
  must use `stage_startup_ticket` so staged cores retain the evidence required
  to come online.
- Removed public startup-epoch accessors and kept startup epochs redacted in
  debug output.
- Added best-effort drop clearing for startup evidence core, hardware ID,
  coordinator, and epoch fields.
- Rejected newline and carriage-return characters in release-manifest dynamic
  path/version fields before writing manifest evidence.
- Added `startup_evidence_ok=true` to the `multicore-topology` serial evidence.
- Moved current image and manifest names to `v0.35.1`.
- Split the release plan so `v0.35.1` is the AP startup evidence milestone.
  The real x86_64 QEMU AP startup path is now tracked after the v0.35.2
  preflight milestone.

## Security Notes

- Direct online-without-startup is no longer available through the public
  `CoreTopology` API.
- Arrival evidence is non-forgeable outside `aesynx-core` because its fields
  are private and the only constructor is ticket-derived.
- Arrival evidence is consumed by `mark_hardware_online`, so repeated use is
  prevented by Rust ownership rather than only by the runtime state machine.
- Startup staging returns the only valid ticket for the staged core; no public
  API remains that can stage a core while silently discarding its evidence.
- Startup evidence must match the epoch recorded on the staged topology entry;
  stale and future epoch evidence fail before topology mutation.
- Mismatched core ID or hardware ID fails before topology mutation.
- Matching checks evaluate both core ID and hardware ID before deciding whether
  to reject, avoiding a simple short-circuit mismatch oracle in the model.
- Role assignment may still occur between startup staging and arrival, matching
  the intended bootstrap flow.
- Release manifest dynamic fields reject newline injection before the manifest
  is written.
- No attacker-unpredictable token consumer was added; QEMU still reports
  `drbg_self_test=false`, and random-token use remains blocked until the DRBG
  implementation exists.

## QEMU Evidence

Expected normal boot evidence includes:

```text
multicore-topology qemu_smp_cores_ok=true hardware_online_ok=true role_assignment_ok=true bootstrap_ok=true scheduler_ok=true driver_service_ok=true idle_ok=true startup_evidence_ok=true multicore_barrier_ok=true
[TEST] multicore-topology=ok
```

The generated manifest name is:

```text
name=Aesynx v0.35.1 AP startup evidence candidate
```

## Non-Claims

- No AP startup trampoline.
- No secondary core executes Rust code yet.
- No per-core GDT, IDT, TSS, or IST setup yet.
- No directed IRQ routing to AP-owned service domains yet.
- No cross-core message fabric yet.

# Aesynx v0.35.0 Release Candidate Notes

`v0.35.0` adds the QEMU multicore topology baseline. The kernel smoke suite now
runs under a four-vCPU QEMU machine and records an explicit no_std topology
model for the bootstrap, scheduler, driver/service, and idle core roles before
real AP startup work begins.

This is intentionally topology and ownership evidence. It does not start
secondary cores yet.

## Changes

- Added `CoreTopology` to `aesynx-core`.
- Added `CoreHardwareState` for discovered, startup-staged, online, and
  quarantined hardware states.
- Added `CoreAssignmentState` so hardware-online state and Aesynx role
  assignment are not conflated.
- Added duplicate hardware-ID rejection for topology entries.
- Added validate-then-commit topology tests for duplicate IDs, failed
  transitions, and epoch overflow.
- Added kernel `multicore_topology_smoke` coverage for four modeled QEMU cores.
- Added `aesynx-core::QEMU_MULTICORE_TOPOLOGY_CORES` so the topology model and
  xtask QEMU `-smp` count are tied to the same constant.
- Added QEMU boot markers:
  - `multicore-topology qemu_smp_cores_ok=true`
  - `hardware_online_ok=true`
  - `role_assignment_ok=true`
  - `bootstrap_ok=true`
  - `scheduler_ok=true`
  - `driver_service_ok=true`
  - `idle_ok=true`
  - `multicore_barrier_ok=true`
  - `[TEST] multicore-topology=ok`
- Changed xtask QEMU smoke runs to launch QEMU with `-smp 4`.
- Added `qemu_smp_cpus=4` to generated image manifests.
- Hardened kernel service queues so each queue records an owner core and
  push/pop/completion/pending-count operations reject non-owner callers before
  mutation.
- Updated QEMU image names and manifest metadata to `v0.35.0`.

## Security Notes

- Topology mutation keeps the existing fail-closed style: validation happens
  before committing new entries or state transitions.
- Duplicate logical core IDs and duplicate hardware IDs are rejected.
- Failed transitions leave the prior topology status and entry state unchanged.
- Topology, registry, and boot-barrier setup mutation is owner-scoped to the
  bootstrap owner core in the current model.
- Service queue mutation and inspection now checks the recorded owner core at
  runtime. This is defense in depth for the current single-owner queues; future
  shared-memory or SMP rings still need real atomic head/tail synchronization
  before they can be treated as cross-core data structures.
- Role assignment is rejected after a core reaches hardware-online state.
- Hardware-online transition now requires the startup-staged state; discovered
  cores cannot jump directly to online.
- Quarantine is a reachable terminal hardware/local state for modeled cores.
- Quarantined cores are excluded from active assignment and role aggregate
  counters so status cannot treat an isolated core as live bootstrap,
  scheduler, driver/service, or idle capacity.
- Unassigned discovered cores are not counted as active idle-role assignments.
- The internal topology mutation epoch is no longer exposed through public
  topology status or debug evidence.
- Invalid x86_64 exception-frame pointers now enter a terminal halt path
  instead of returning toward `iretq` with untrusted frame state.
- Live interrupt-gate installation remains crate-private and boot/smoke-only;
  a future NMI exclusion strategy is required before live IDT mutation can
  become a general runtime facility.
- Added a strict CPU-hardening plan constructor that fails closed unless NX,
  SMEP, SMAP, and UMIP are all available; the QEMU default policy remains
  compatibility-oriented and enables SMEP/SMAP/UMIP only when CPUID reports
  support.
- Entropy policy now distinguishes raw hardware entropy from DRBG-backed random
  tokens. `RDRAND`/`RDSEED` evidence can classify seed material, but
  `require_random_tokens()` stays disabled until a DRBG path passes its own
  self-test.
- Kernel heap locking rejects reentrant acquisition instead of spinning
  forever, restores the prior interrupt state on the rejected path, and
  production `GlobalAlloc::dealloc` fail-stops on reentrant heap-lock errors.
- Kernel heap allocation accounting now saturates at zero on free-side
  underflow instead of wrapping.
- Production capability root minting rejects zero object generations before
  insertion or audit emission.
- `aesynx-sync::LockRank` now includes an explicit `CoreTopology` rank for
  future topology mutation paths.
- `v0.35.0` does not remove the existing `smp` compile-time tripwires around
  static GDT/TSS/IDT, heap backing, and page-table activation storage.
- QEMU `-smp 4` is now an environment contract for smoke evidence, not proof
  that APs execute kernel code.
- Future AP startup work must keep topology mutation owner-scoped, define the
  AP watchdog/recovery story for permanent quarantine, and replace any
  remaining shared single-core statics before relying on concurrent mutation.

## Non-Goals

- No AP startup trampoline.
- No secondary core executes Rust code.
- No per-core GDT/IDT/TSS/IST installation.
- No cross-core message fabric.
- No directed IRQ routing.
- No scheduler migration between cores.

## Expected QEMU Evidence

The normal boot smoke must include:

```text
multicore-topology qemu_smp_cores_ok=true hardware_online_ok=true role_assignment_ok=true bootstrap_ok=true scheduler_ok=true driver_service_ok=true idle_ok=true multicore_barrier_ok=true
[TEST] multicore-topology=ok
```

The manifest must include:

```text
name=Aesynx v0.35.0 QEMU multicore topology candidate
qemu_smp_cpus=4
```

## Verification

Before pentest:

```bash
cargo fmt --all --check
cargo test -p aesynx-core
cargo test -p aesynx-kernel multicore_topology_smoke
cargo test -p xtask image
scripts/checks.sh
cargo xtask qemu-suite
```

Before tagging `v0.35.0`:

- Generate the SBOM.
- Add the pentest report at `security/pentest/v0.35.0.md`.
- Run `cargo xtask release-ready v0.35.0` after the pentest report, SBOM, and
  release evidence are committed.

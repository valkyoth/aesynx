# Aesynx v0.34.0 Release Candidate Notes

`v0.34.0` adds the first AMP core data structures. The goal is to make core
identity, role, capability metadata, local state, per-core registry ownership,
local telemetry, and boot-barrier state explicit before x86_64 multicore
bring-up begins in `v0.35.0`.

This release does not start secondary cores. It keeps the existing SMP hardware
tripwires in place and only proves the single bootstrap core is represented by
the same model future cores must use.

## Scope

- Added the `aesynx-core` crate.
- Added `CoreRole` for bootstrap, scheduler, driver/service, and idle roles.
- Added heterogeneous capability metadata with ISA and performance-class
  fields plus local-timer, IPI, directed-IRQ, and shared-atomic flags.
- Added `CoreLocal` with explicit `CoreState` and local telemetry counters.
- Added an owner-scoped fixed-capacity `CoreRegistry`.
- Added a fixed-capacity sealed `BootBarrier`.
- Added kernel `amp_core_smoke` coverage and QEMU markers before terminal boot
  success.
- Updated QEMU image names and manifest metadata to `v0.34.0`.

## Security Notes

- `aesynx-core` is safe Rust and `#![forbid(unsafe_code)]`.
- The registry and boot barrier are local no_std models. They are not yet
  atomic SMP data structures and do not claim cross-core synchronization.
- Mutable model state has an explicit owner core. Future shared mutation still
  requires the v0.33.1 concurrency policy and a documented synchronization or
  fabric boundary.
- Core capability metadata is descriptive policy input only. It does not prove
  hardware capabilities until populated from trusted topology discovery in a
  later release.
- The boot barrier is validate-then-commit: failed participant admission and
  unknown-core arrival do not mutate arrival state.
- This release does not enable multicore execution, AP startup, per-core GDT,
  per-core IDT, per-core TSS, IRQ routing, or cross-core fabric messaging.

## QEMU Evidence

The normal boot smoke now requires:

- `amp-core bootstrap_role_ok=true`
- `capabilities_ok=true`
- `registry_ok=true`
- `telemetry_ok=true`
- `barrier_ok=true`
- `[TEST] amp-core=ok`

This proves the bootstrap core is represented as a `CoreLocal` with
bootstrap-role assignment, x86_64 capability metadata, owner-scoped registry
entry, local telemetry, and sealed boot-barrier arrival.

## Verification

Before requesting pentest review, run:

```bash
cargo fmt --all --check
cargo test -p aesynx-core
cargo test -p aesynx-kernel amp_core_smoke
cargo test -p xtask image
scripts/checks.sh
cargo xtask qemu-suite
```

Before tagging `v0.34.0`:

- Confirm GitHub CI and CodeQL are green for the exact commit.
- Add the pentest report at `security/pentest/v0.34.0.md`.
- Run `cargo xtask release-ready v0.34.0` after the pentest report, SBOM, and
  release notes are present.

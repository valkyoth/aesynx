# Aesynx v0.20.0 Release Candidate Notes

Status: implementation candidate; awaiting pentest, GitHub checks, generated
SBOM, release-readiness validation, signed tag, and push.

## Scope

`v0.20.0` adds the first kernel-facing capability table. The table is still a
bounded software authority model, but normal QEMU boot now proves create, check,
derive, revoke, generation, and redacted telemetry behavior after Aesynx has
loaded its own CR3 root.

## Delivered So Far

- Added `CapabilityTable<const SLOTS: usize>` to `aesynx-cap`.
- Table storage is fixed-capacity, `no_std`, no-alloc, and safe Rust.
- Root capabilities can be inserted into vacant slots with checked external
  `CapId` creation.
- `CapId` lookup validates slot bounds and slot generation before returning
  authority.
- Permission checks reject missing authority without exposing capability
  internals.
- Derive and grant table operations route through audited capability methods.
- Revocation checks `REVOKE` authority before committing and removes all table
  entries for the target object in one preflighted operation.
- Revocation is documented as total in v0.20: the authority capability used for
  the revoke is invalidated with every other in-table capability for that
  object.
- Revoked slots increment their slot generation so old `CapId` values fail
  closed as stale.
- Derive/grant table operations now preflight table capacity and output `CapId`
  construction before emitting an audit event, preventing phantom successful
  audit records when the table is full.
- `RevocationEpochStore` now exposes `revoke_object_live`, requiring callers to
  supply current generation and epoch evidence so stale or already-revoked
  authorities cannot increment an object's revocation epoch.
- Large heap page-run allocations are zeroed before first return, not only on
  reuse after free.
- Capability-table `Debug` reports only aggregate capacity and occupancy.
- Added host tests for root insertion, permission checks, audited derivation,
  stale-id rejection after revoke, and full-table failure.
- Added a post-CR3 kernel smoke that exercises root creation, checked read
  authority, audited cross-owner derivation, child write denial, revoke, stale
  root/child `CapId` rejection, aggregate redacted status, and `[TEST] cap=ok`.
- QEMU marker contracts, image names, and image manifest metadata now track the
  `v0.20.0` kernel capability-table candidate.

## Non-Claims

- No syscall, IPC, object registry, memory-map service, driver, package, or
  userspace operation is protected by runtime capabilities yet.
- The capability table is fixed-capacity and single-kernel-smoke scoped; it is
  not yet per-task, per-principal, SMP-safe, persistent, or dynamically grown.
- Revocation removes matching in-table object capabilities, but there is no
  cross-core stale-cache invalidation, revoke broadcast, persistent epoch
  database, or object registry yet.
- Total revoke also removes the authority capability used for the revoke. A
  future object registry or epoch store must re-mint post-revoke authority when
  that behavior is needed.
- Slot generations fail closed instead of wrapping. A future persistent table
  must rebuild or retire slots before any slot reaches `u32::MAX`.
- `CapId` is still an index plus generation identifier. It is not authority by
  itself and is not an entropy-backed bearer token.
- Future entropy-backed token or capability-id generation must use
  `require_random_tokens()` as the policy gate. Direct `primary_source` reads are
  telemetry/classification only until a grep lint or equivalent test enforces
  the call pattern.
- No ring-3 process isolation or authenticated call path exists yet.

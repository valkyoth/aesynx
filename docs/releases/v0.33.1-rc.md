# Aesynx v0.33.1 Release Candidate Notes

`v0.33.1` adds the first explicit concurrency discipline before multicore work
begins. Aesynx uses x86_64 SMP/APIC mechanisms for hardware bring-up, but the
target architecture is software-defined AMP and a multikernel fabric: explicit
core roles, per-core ownership, directed IRQ ownership, and bounded messages
instead of a classic shared-everything SMP kernel. The scope is deliberately
narrow: safe no_std synchronization model primitives, host tests for guard
behavior, QEMU evidence that the primitives compile into the kernel image, and
documentation that future shared-state work must follow.

## Scope

- Added the `aesynx-sync` crate.
- Added `LocalInterruptMask` and `InterruptGuard` with previous-state restore.
- Added nested interrupt-guard behavior where inner guards do not re-enable
  interrupts that an outer guard masked.
- Added `EarlyLock` with guard-owned release semantics and no public bare
  unlock path.
- Added `try_lock_irq()` so early locks can be acquired with local interrupts
  masked and restored in the right order.
- Added `LockRank`, `LockOrderTracker`, and `LockOrderGuard` for conservative
  lock-order validation.
- Added kernel `concurrency_smoke` coverage and QEMU markers before terminal
  boot success.
- Added `docs/concurrency-policy.md`.
- Clarified the future capability-based shared-memory-window model: shared
  buffers are object-backed capabilities, not raw physical authority, with
  read-only sealing preferred for zero-copy assets and writable sharing gated by
  explicit synchronization policy.
- Updated QEMU image names and manifest metadata to `v0.33.1`.

## Security Notes

- `aesynx-sync` is safe Rust and `#![forbid(unsafe_code)]`.
- The new primitives are model/early-boot primitives, not a full production
  shared-kernel SMP scheduler lock framework.
- The current kernel is still single-core. The existing `smp` tripwires remain.
- Future milestones that say SMP must distinguish x86_64 hardware bring-up from
  shared-kernel architecture; the Aesynx target is AMP/multikernel after cores
  are online.
- Interrupt guards restore previous interrupt state instead of blindly enabling
  interrupts.
- Lock-order tracking rejects equal-rank and lower-rank acquisition while a
  higher rank is held.
- Code holding locks must not block, allocate, log, call out to drivers, call
  into AI policy, or invoke external callbacks unless a fatal halt path
  documents a narrow exception.
- Shared service-queue slot vacate paths must scrub payload storage before
  cross-domain observation in any future multi-core/shared-memory queue.
- Capability-based shared memory remains planned, not implemented in this
  release. The roadmap now requires revocation, audit, TLB shootdown, and mapper
  alias-policy work before shared buffers can become live.

## QEMU Evidence

The normal boot smoke now requires:

- `concurrency irq_guard_ok=true`
- `nested_irq_guard_ok=true`
- `early_lock_ok=true`
- `irq_lock_ok=true`
- `lock_order_ok=true`
- `[TEST] concurrency=ok`

This proves the guard and lock model compiles into the kernel image and remains
green in the current single-core QEMU boot path.

## Verification

Before requesting pentest review, run:

```bash
cargo fmt --all --check
cargo test -p aesynx-sync
cargo test -p aesynx-kernel concurrency_smoke
cargo test -p xtask image
scripts/checks.sh
cargo xtask qemu-suite
```

Before tagging `v0.33.1`:

- Confirm GitHub CI and CodeQL are green for the exact commit.
- Add the pentest report at `security/pentest/v0.33.1.md`.
- Run `cargo xtask release-ready v0.33.1` after the pentest report, SBOM, and
  release notes are present.

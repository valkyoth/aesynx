# Aesynx v0.37.0 Release Candidate Notes

`v0.37.0` integrates the pairwise IPC fabric with the capability and object
authority models. The kernel smoke grants a READ capability from a sender table
into a receiver table, routes the receiver-allocated `CapId` through a stamped
`GrantCap` IPC message, and proves revocation reaches the receiver by bumping
the object registry revocation epoch.

This is still model-backed evidence. Secondary cores do not execute Aesynx Rust
code yet, the pairwise queues are not live hardware atomics, and there is no
userspace endpoint or production revoke broadcast protocol. The value of this
release is the authority-transfer shape and the fail-closed checks later live
fabric work must preserve.

## Changes

- Added `CapabilityTable::grant_to_table_with_audit`, allowing a sender table
  to allocate a granted capability directly in a receiver table after live
  generation/epoch validation, source permission checks, and audit acceptance.
- Added object-registry support for `RevocationEpochStore`, so the kernel
  object registry can be the canonical epoch source for cross-table revocation
  tests.
- Added a kernel cap-IPC smoke that:
  - creates a memory object and root authority,
  - grants READ authority into a receiver table,
  - sends the receiver `CapId` over a kernel-stamped `GrantCap` message,
  - rejects receiver WRITE access,
  - rejects sender grant attempts from a cap without `GRANT`,
  - sends a `RevokeCap` notification,
  - bumps the object registry revocation epoch, and
  - proves the receiver cap resolves as revoked.
- Updated QEMU boot smoke markers, image names, and manifests to the
  `v0.37.0` candidate.

## Security Notes

- Receiver `CapId` allocation is validate-before-commit: target capacity and ID
  construction are checked before audit and table mutation.
- Missing sender `GRANT` permission leaves the receiver table unchanged and does
  not emit a phantom grant audit event.
- Cross-owner grant semantics continue to strip `GRANT`, `REVOKE`, and `ADMIN`
  from the receiver capability.
- Revoke evidence is tied to the object registry revocation epoch, not only to a
  local sender-table revoke. Receiver capabilities for the same object fail live
  object resolution after the epoch bump.
- Revocation epoch increments fail instead of wrapping and leave registry state
  unchanged on overflow.
- The IPC message carries a redacted `MessagePayload::Cap` value in diagnostics;
  `CapId` remains an opaque identifier in the serial evidence.

## Non-Goals

- No secondary core executes Aesynx code.
- No APIC IPI delivery path.
- No live shared-memory queue between hardware cores.
- No userspace endpoint, syscall ABI, or blocking receive path.
- No production distributed revoke broadcast or consensus protocol.
- No capability-bearing slot scrubbing beyond the existing model warning for
  future cross-domain shared-memory rings.

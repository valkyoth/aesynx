# Aesynx v0.35.2 Release Candidate Notes

`v0.35.2` adds AP startup preflight evidence. It does not start secondary cores
yet; it proves the future AP trampoline has a typed, fail-closed resource gate
and that the current shared-bootstrap descriptor-table state blocks execution.

## What Changed

- Added `ApStartupPreflight` to `aesynx-core`.
- Added `ApStartupResource` records for staged cores.
- Added explicit descriptor-table readiness with
  `SharedBootstrapOnly` and `PerCoreReady` states.
- Required preflight resources to target topology entries that are already
  `StartupStaged` and locally `Booting`.
- Required page-aligned AP stack ranges of at least 16 KiB.
- Required a nonzero startup watchdog tick budget for every planned AP.
- Rejected duplicate logical core IDs, duplicate hardware IDs, overlapping
  startup stacks, non-owner callers, unstaged entries, undersized stacks, and
  missing watchdogs before mutation.
- Added QEMU smoke evidence for `ap_preflight_ok=true` and
  `ap_execution_blocked_ok=true`.
- Hardened kernel heap free-list membership checks so corrupt valid cycles are
  bounded and reported as `CorruptFreeList` instead of looping while interrupts
  are masked.
- Moved heap free-side byte accounting to a compare-exchange saturating
  decrement so the operation does not rely on a load/store pair when the heap
  later migrates toward per-core or SMP-aware ownership.
- Replaced startup ticket/arrival drop clearing through `black_box` with atomic
  field storage and release-store clearing on drop.
- Added the `strict-cpu-hardening` feature selector so high-assurance builds can
  choose fail-closed CPU hardening while default QEMU builds remain compatible.
- Expanded the release plan with explicit blockers for Spectre-class controls,
  KASLR/PIE, hardware entropy reads, DRBG implementation, guarded per-core
  double-fault IST stacks, and migration away from shared bootstrap descriptor
  statics before AP execution.
- Moved current image and manifest names to `v0.35.2`.
- Split the release plan so `v0.35.2` is AP startup preflight and the real
  x86_64 QEMU AP startup path is tracked as `v0.35.3`.

## Security Notes

- AP launch resources are validate-then-commit. Failed preflight additions do
  not partially install AP stack or hardware-ID records.
- Shared bootstrap-only descriptors are represented as an explicit blocker, not
  an implicit TODO. The QEMU smoke requires execution to remain blocked while
  descriptor tables are not per-core ready.
- The preflight API is owner-scoped, matching the existing topology ownership
  discipline.
- The model keeps AP startup stack windows unique before any future trampoline
  can rely on them.
- Startup evidence fields are stored in atomics and cleared through release
  stores on drop; this avoids relying on optimizer-sensitive best-effort
  clearing.
- Heap free-list corruption now remains visible through the existing corrupt
  free-list telemetry even when the allocator fails closed instead of halting
  immediately.
- Strict CPU hardening is available as an opt-in build feature. It is not the
  default because current QEMU/general development targets may not expose every
  optional hardening bit.
- No attacker-unpredictable token consumer was added; QEMU still reports
  `drbg_self_test=false`, and random-token use remains blocked until the DRBG
  implementation exists.
- Spectre-class CPU controls, KASLR, hardware entropy instruction reads, DRBG
  construction, per-core descriptor ownership, and double-fault IST guard pages
  remain planned blockers before multi-domain or real-hardware deployment
  claims.

## QEMU Evidence

Expected normal boot evidence includes:

```text
multicore-topology qemu_smp_cores_ok=true hardware_online_ok=true role_assignment_ok=true bootstrap_ok=true scheduler_ok=true driver_service_ok=true idle_ok=true startup_evidence_ok=true ap_preflight_ok=true ap_execution_blocked_ok=true multicore_barrier_ok=true
[TEST] multicore-topology=ok
```

The generated manifest name is:

```text
name=Aesynx v0.35.2 AP startup preflight candidate
```

## Non-Claims

- No AP startup trampoline.
- No secondary core executes Rust code yet.
- No per-core GDT, IDT, TSS, or IST setup yet.
- No directed IRQ routing to AP-owned service domains yet.
- No cross-core message fabric yet.

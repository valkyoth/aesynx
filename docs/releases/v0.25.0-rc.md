# Aesynx v0.25.0 Release Candidate Notes

Status: release candidate

## Summary

`v0.25.0` adds the host-side service queue model that will guide the kernel
service queues in `v0.26.0`. The model keeps queue behavior small, explicit,
and testable before it becomes part of the no_std kernel IPC path.

## Added

- Added `models/aesynx-ipc-model`.
- Added explicit service kinds for log, timer, object, capability, memory,
  driver, and telemetry services.
- Added `ServiceRequest` with nonzero message IDs, verified caller core,
  target service, message kind, and bounded IPC payload.
- Added `ServiceCompletion` with request identity, status, and response
  payload.
- Added `ServiceRingQueue<T>` as a fixed-capacity FIFO model with wraparound.
- Added queue ordering evidence for release publication and acquire
  observation.
- Added object-record revocation epoch tracking so registry capability
  resolution can enforce both generation and epoch freshness.
- Added `LiveAuthorityView` and object-registry-backed live authority state so
  table-level derive, grant, and revoke operations validate against current
  object generation/epoch before audit emission.
- Added redacted debug output for IPC headers, requests, payloads, inline
  payload buffers, service requests, and service completions.
- Enabled release overflow checks and explicit aborting release panics at the
  workspace profile level.
- Added an intentional kernel `smp` feature compile-time tripwire while the
  heap backing store and activation arena/stack remain single-core statics.

## Security And Correctness

- Zero message IDs are rejected for modeled service requests.
- Zero-capacity queues are rejected.
- Full queue pushes fail before mutating queue state.
- Empty queue pops fail before mutating queue state.
- FIFO ordering is preserved across ring wraparound.
- Queue publish/observe tests lock in the intended release/acquire contract
  before the design moves into kernel-owned queues.
- The ring queue uses explicit wrapping arithmetic for index advancement to
  document overflow behavior even though a `usize::MAX` host queue is not
  practically constructible.
- Capability audit documentation now states that audit sinks must not reenter
  the table they are auditing.
- Security controls now track pre-SMP replacement of `static mut` kernel heap
  and activation storage, bounded IRQ-masked heap latency before material heap
  growth, hardware entropy self-test requirements, and serialization redaction
  requirements.
- Object-registry capability resolution rejects stale revocation epochs, so a
  capability revoked by epoch bump cannot pass object resolution merely because
  its object generation still matches.
- Capability table derive, grant, and revoke paths no longer perform
  tautological self-checks against the capability's own generation/epoch.
  Callers must provide a live authority view, and stale live state rejects
  before audit emission or table mutation.
- IPC/service debug output redacts inline payload bytes, capability/object IDs,
  core IDs, message IDs, request IDs, and service payloads to prevent logging
  from becoming an authority or data leak.
- Queue ordering evidence construction is no longer public/free-form; callers
  can observe the release/acquire service-queue contract but cannot construct
  arbitrary relaxed evidence through the public API.
- Generation-exhausted object slots remain live instead of wrapping the
  generation counter; security controls now require future diagnostics to
  surface that condition as a registry-retirement signal.
- Capability documentation now states that unscoped parent capabilities may
  derive bounded children subject to permission checks, while real object-range
  enforcement belongs at the object, memory, or syscall layer.
- The v0.26.0 service-queue plan now requires typed validation of
  `MessageRequest::dst` before it can be used as a queue/core selector.

## Known Limitations

- The queue is a host-side model only; v0.25.0 does not add kernel service
  queues, wakeups, blocking receives, syscall entry, scheduler integration, or
  SMP producer/consumer paths.
- The model records intended acquire/release semantics as evidence. It does
  not yet use hardware atomics or shared-memory queue slots.
- The QEMU serial marker set is unchanged from `v0.24.0`; this release prepares
  the service queue design for the next kernel milestone.

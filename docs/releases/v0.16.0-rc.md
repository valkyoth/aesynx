# Aesynx v0.16.0 Release Candidate Notes

Status: implementation candidate; pentest and GitHub checks are still required
before tagging.

## Scope

`v0.16.0` adds the first explicit kernel mapping policy descriptor on top of
the bounded `aesynx-mm` page-table mapper model. The normal boot smoke derives
kernel text, rodata, and data/BSS ranges from page-granular linker section
symbols, then verifies the shape Aesynx expects for early kernel memory before
future CR3 activation work:

- Kernel text is read/execute, write-protected, kernel-space, normal memory,
  and local.
- Kernel rodata is read-only, non-executable, kernel-space, normal memory, and
  local.
- Kernel data/BSS is read/write, non-executable, kernel-space, normal memory,
  and local.
- Reserved heap and guard-page windows are canonical high-half ranges that
  remain unmapped.
- Page zero remains unmapped.
- Policy ranges are non-empty, checked for overflow, and non-overlapping.

This is still a policy model and QEMU smoke gate. It does not replace Limine's
active CR3, does not reclaim bootloader page tables, and does not yet prove live
hardware faults for writes to text, execution from NX data, or guard-page
touches.

## Deliverables

- `aesynx-mm::KernelVirtualRange`.
- `aesynx-mm::KernelMappingPolicy`.
- `aesynx-mm::KernelMappingPolicyReport`, including verifier-derived status
  fields consumed by the QEMU paging-policy smoke.
- `PageTableMapper::verify_kernel_mapping_policy`.
- Linker symbols for text, rodata, and data/BSS start/end boundaries.
- Safe `KernelSectionLayout` values produced by the reviewed linker-symbol
  boundary.
- Safe `KernelMappingPlan` derivation in the kernel library, with host tests
  for section ordering, page alignment, canonical high-half section and
  reservation ranges, reserved-range presence, and overflow.
- Redacted debug output for kernel virtual ranges.
- Normal boot kernel mapping smoke after the v0.15 page-table smoke.
- New serial status line:

```text
paging-policy mapped_pages=<n> reserved_pages=<n> text_pages=<n> rodata_pages=<n> data_pages=<n> section_layout_ok=true text_rx_ok=true rodata_read_only_ok=true data_rw_nx_ok=true heap_reserved_ok=true guard_page_ok=true null_page_ok=true
```

- New normal boot marker:

```text
[TEST] paging-policy=ok
```

## Security Hardening and Fixes Since v0.15.0

- Kernel text policy now requires the exact `ReadExecute` kernel mapping flags,
  write protection, executable state, normal memory, and local mapping state.
- Rodata policy now requires the exact `ReadOnly` kernel mapping flags,
  write protection, non-executable state, normal memory, and local mapping
  state.
- Data/BSS policy now requires the exact `ReadWrite` kernel mapping flags,
  non-executable state, normal memory, and local mapping state.
- Reserved heap and guard-page policy now requires unmapped high-half ranges,
  so future heap activation can start from an explicit reservation rather than
  accidental free space.
- Null-page policy now requires exactly one unmapped page at virtual address
  zero.
- Kernel policy range validation rejects zero-page ranges, range arithmetic
  overflow, and overlapping text/rodata/data/heap/guard/null ranges before
  reporting a successful policy check.
- Normal boot now consumes page-granular linker section boundaries instead of
  hardcoded synthetic section sizes, so the policy smoke tracks the kernel
  image layout that Limine loads.
- Section-layout planning now lives in host-testable safe kernel-library code
  instead of only in the freestanding QEMU smoke path. This gives normal unit
  coverage for malformed section order, unaligned boundaries, low-half and
  noncanonical section layouts, empty heap/guard reservations, and arithmetic
  overflow.
- The linker-derived kernel mapping plan now fails closed before policy
  creation when text, rodata, data/BSS, reserved heap, or guard ranges are
  outside canonical high-half kernel virtual space.
- Policy verification starts with mapper audit, so corrupted table/index state
  fails closed before permission policy is evaluated.
- Host tests cover writable text rejection, executable data rejection, mapped
  guard-page rejection, mapped reserved-heap rejection, mapped null-page
  rejection, writable/executable rodata rejection, user-accessible text
  rejection, device/global kernel-data rejection, bad null-page descriptors,
  low-half reserved ranges, unaligned and noncanonical policy ranges,
  zero-page ranges, overflowing ranges, overlapping policy ranges, redacted
  kernel range/policy/report debug output, low-half/noncanonical section-plan
  rejection, reserved-range overflow rejection, and kernel mapping plan
  derivation.
- QEMU boot success now requires both the page-table mapper marker and the new
  paging-policy status/ok markers.
- The paging-policy serial booleans are read from the `aesynx-mm` policy
  report instead of being hardcoded in the freestanding smoke path.
- The policy report fields are produced by named verifier sub-checks for text,
  rodata, data/BSS, reserved heap, guard page, and null page.
- Image manifests and xtask marker tests now include the paging-policy markers
  for release traceability.

## Verification

- `cargo test -p aesynx-mm`.
- `cargo test -p xtask`.
- `cargo xtask qemu-suite`.
- Pentest follow-up before tag.
- GitHub CI green before tag.

## Non-Claims

- No active CR3 replacement.
- No production page-table ownership.
- No live hardware write-to-text, execute-NX, or guard-page fault proof.
- No heap allocator activation.
- No user address spaces.
- No page-fault recovery path.

# Aesynx v0.27.0 Release Candidate Notes

Status: implementation candidate; pending pentest and GitHub checks before tag.

## Summary

`v0.27.0` adds the kernel task model. The existing `aesynx-sched` crate now has
split task and queue modules, fixed-capacity local run queues, fixed-capacity
wait queues, checked state transitions, and redacted task debug output. The
kernel smoke creates multiple task objects after CR3 activation and verifies
run/wait queue behavior before reporting `[TEST] task-model=ok`.

## Added

- Split `aesynx-sched` into `task` and `queue` modules so the scheduler model
  does not grow into one large file.
- Added `LocalRunQueue` with fixed-capacity FIFO behavior for runnable tasks on
  one owner core.
- Added `WaitQueue` and `WaitReason` for message, timer, and object waits.
- Added `QueueStatus` and `TaskQueueError` so callers can inspect aggregate
  queue state without exposing raw queue storage.
- Added `Task` debug redaction for task IDs.
- Added a kernel task-model smoke that creates multiple task objects, routes
  runnable tasks through the local run queue, moves tasks into message and
  timer wait queues, wakes one message waiter back to runnable state, and
  verifies wrong-core and zero-task-ID admission failures.
- Added QEMU marker coverage for `task-model created=` and
  `[TEST] task-model=ok`.

## Security And Correctness

- Queue construction rejects zero-capacity queues.
- Queue admission rejects zero task IDs.
- Local run queues reject tasks owned by another core.
- Local run queues reject non-runnable tasks.
- Duplicate task IDs are rejected before queue mutation.
- Full queue pushes and empty pops fail before mutating queue state.
- Wait queues accept only tasks whose state matches the queue's wait reason.
- Wait wakeup uses the checked task transition API to return a task to
  `Runnable`.
- Task debug output redacts task IDs so future logs do not accidentally expose
  scheduler identities as stable authority handles.

## Known Limitations

- This is a data model and smoke path, not a real executor.
- There is no context switch, saved register frame, kernel stack per task,
  preemption, timeslice accounting, syscall ABI, userspace task object, or SMP
  scheduler yet.
- The run queue is single-core and fixed-capacity. Future SMP work must move
  scheduler queues into explicit per-core ownership with IRQ-safe locking and
  lock-ordering rules.
- Wait queues do not yet bind to real IPC, timer, object, or capability events.

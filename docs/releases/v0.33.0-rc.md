# Aesynx v0.33.0 Release Candidate Notes

`v0.33.0` extends the advisory AI policy interface with the first scheduler
policy model that can make a bounded, deterministic decision without loading or
trusting an AI model. The goal is to keep the future AI pathway structurally
present while preserving fail-closed, fixed-point, inspectable behavior.

## Scope

- Added `HeuristicScheduleScore`, a bounded `0..=10_000` fixed-point score.
- Added `SchedulerPolicyConfig` with an explicit heuristic enable switch,
  fallback core, heuristic target core, and minimum score threshold.
- Added `HeuristicSchedulerPolicy` as the scheduler-domain `PolicyEngine`.
- Added `SchedulerPolicyDecisionRecord` so scheduler policy choices can be
  inspected without exposing authority-bearing task identity.
- Added `score_features()` for bounded scheduler features:
  `run_queue_len`, `ipc_depth`, `queue_pressure`, `object_locality_score`,
  `idle_ratio`, `migration_cost`, and compact `priority`.
- Kept the existing scheduler feature validator as the admission gate for every
  fixed-point feature field.
- Updated the kernel AI policy smoke to prove both disabled-fallback behavior
  and enabled heuristic evidence.
- Updated QEMU image names and manifest metadata to `v0.33.0`.

## Security Notes

- The scheduler heuristic is not AI inference, not model loading, and not
  online learning.
- The policy remains advisory. It does not grant capabilities, change task
  ownership, bypass scheduler queue checks, or make a final authority decision.
- The disabled configuration returns deterministic zero-confidence local
  fallback, matching the prior local round-robin policy shape.
- Invalid out-of-range scheduler features fail closed with
  `DecisionReason::SafetyRejected` and deterministic fallback.
- Low heuristic scores fall back instead of selecting the heuristic target.
- The heuristic decision carries no model ID and no raw task identifier.
- Hashes and signatures in model manifests are still metadata-presence gates,
  not cryptographic verification or a trust boundary.

## QEMU Evidence

The normal boot smoke now requires:

- `ai-policy schema=1`
- `manifest_metadata_gate_ok=true`
- `heuristic_enabled=true`
- `heuristic_score=`
- `heuristic_disabled_fallback_ok=true`
- `[TEST] ai-policy=ok`

The smoke path verifies that a safe scheduler manifest is accepted, a manifest
without required fallback is rejected, the heuristic can be disabled for
zero-confidence local fallback, and the enabled heuristic produces a bounded
score for the configured heuristic core.

## Verification

Before requesting pentest review, run:

```bash
cargo fmt --all --check
cargo test -p aesynx-ai-policy
cargo test -p aesynx-kernel ai_policy_smoke
cargo test -p xtask image
scripts/checks.sh
cargo xtask qemu-suite
```

Before tagging `v0.33.0`:

- Confirm GitHub CI and CodeQL are green for the exact commit.
- Add the pentest report at `security/pentest/v0.33.0.md`.
- Run `cargo xtask release-ready v0.33.0` after the pentest report, SBOM, and
  release notes are present.

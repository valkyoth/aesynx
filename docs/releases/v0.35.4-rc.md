# Aesynx v0.35.4 Release Candidate Notes

`v0.35.4` starts closing the multi-domain hardening blockers that must be
resolved before real AP execution, ring-3 userspace, or real-hardware deployment
claims. The first release slice makes Spectre-class CPU controls explicit in
the x86_64 hardening model.

## What Changed

- Added CPUID-gated speculative-execution capability tracking for:
  `IBRS/IBPB`, `STIBP`, `SSBD`, and `ARCH_CAPABILITIES`.
- Admitted the reviewed `IA32_SPEC_CTRL` MSR in the x86_64 CPU hardening
  boundary.
- The default CPU hardening plan now requests supported `IA32_SPEC_CTRL` bits
  for IBRS, STIBP, and SSBD.
- The strict CPU hardening plan now fails closed when speculative controls or
  `ARCH_CAPABILITIES` evidence are missing.
- CPU hardening read-back verification now includes requested SPEC_CTRL bits.
- QEMU CPU-hardening evidence remains redacted boolean-only and now reports:
  `ibrs`, `ibpb_supported`, `stibp`, `ssbd`, and `arch_capabilities`.
- Updated image names and generated manifests to `v0.35.4`.
- Updated documentation to keep AP startup, KASLR, DRBG, and remaining
  hardening items visible as non-claims.

## Security Notes

- Raw MSR values are not logged. Serial output exposes only booleans.
- `IA32_SPEC_CTRL` is read or written only when CPUID reports at least one
  supported speculative-control bit.
- Default QEMU/general builds stay capability-gated and do not fail just because
  a virtual CPU lacks optional speculative controls.
- High-assurance builds using `strict-cpu-hardening` now require NX, SMEP, SMAP,
  UMIP, IBRS/IBPB, STIBP, SSBD, and `ARCH_CAPABILITIES`.
- This release makes the speculative-control path explicit; it does not claim
  that every microarchitectural side channel is solved.

## QEMU Evidence

Expected normal boot evidence includes:

```text
cpu-hardening nx=<bool> wp=<bool> smep=<bool> smap=<bool> umip=<bool> ibrs=<bool> ibpb_supported=<bool> stibp=<bool> ssbd=<bool> arch_capabilities=<bool>
[TEST] cpu-hardening=ok
```

The generated manifest name is:

```text
name=Aesynx v0.35.4 hardening blockers candidate
```

## Non-Claims

- No AP startup trampoline.
- No secondary core executes Rust code yet.
- No KASLR/PIE implementation yet.
- No DRBG implementation yet; `random_tokens_available=false` remains the only
  acceptable production state until DRBG self-tests pass.
- No complete NMI-safe live-IDT mutation implementation yet.
- No final arch-backed IRQ-disable proof token yet.
- No final root-capability bootstrap token enforcement yet.
